# Repo Tutor 使用指南

## 目标用户

本指南面向本地单用户使用 Repo Tutor 的开发者或学习者。

Repo Tutor 当前适合做这些事：

- 输入一个本地仓库绝对路径，快速得到首轮阅读报告。
- 输入一个公开 GitHub 仓库 URL，先看结构、入口候选和阅读起点。
- 在首轮报告之后继续追问入口、模块、主流程、依赖和目录职责。
- 在同一会话里逐步建立对仓库结构的整体理解。

## 当前能力边界

- 第一版只维护一个活跃会话。
- 仓库分析是只读的，不会修改目标仓库文件。
- M1-M4 的确定性模块负责只读仓库分析，不依赖 LLM。
- 首轮报告和多轮聊天由 M6 在线 LLM 生成；M1-M4 分析结果会通过 LLM 工具上下文提供给模型作为参考。
- 静态分析结果不是唯一真相。回答中涉及入口、流程、分层等内容时，应把它们看作候选或参考，证据不足处会标注不确定性。
- 敏感文件可能会被标记为存在，但正文不会进入分析结果、SSE、日志或提示词。

## 大模型 Key 配置

当前版本不再从环境变量读取大模型 Key。

请直接编辑仓库根目录下的 `llm_config.json`：

```json
{
  "api_key": "your_key_here",
  "base_url": "https://api.deepseek.com",
  "model": "deepseek-chat",
  "timeout_seconds": 60
}
```

字段说明：

- `api_key`：必填，大模型 API Key
- `base_url`：可选，兼容 OpenAI SDK 的服务地址
- `model`：可选，模型名
- `timeout_seconds`：可选，单次调用超时时间

默认值：

- `base_url` 默认是 `https://api.deepseek.com`
- `model` 默认是 `deepseek-chat`
- `timeout_seconds` 默认是 `60`

如果 `llm_config.json` 不存在、不是合法 JSON，或 `api_key` 为空，M6 调用会直接报错。

## 启动方式

### 后端

推荐方式之一：

```bash
uv sync --extra dev
uv run uvicorn backend.main:app --reload --host 127.0.0.1 --port 8000
```

Windows 也可以直接使用：

```cmd
scripts\dev_backend.cmd
```

### 前端

```bash
cd frontend
npm install
npm run dev
```

Windows 也可以直接使用：

```cmd
scripts\dev_frontend.cmd
```

### 访问地址

- 前端开发地址：`http://127.0.0.1:5173`
- 后端开发地址：`http://127.0.0.1:8000`

前端开发服务器已配置 `/api` 代理到后端，因此浏览器里只需要打开前端地址。

## 第一次使用

1. 先启动后端，再启动前端。
2. 在浏览器打开 `http://127.0.0.1:5173`。
3. 在输入框中填入以下任一种内容：

- 本地绝对路径，例如 `C:\repo\demo`
- 公开 GitHub 仓库 URL，例如 `https://github.com/owner/repo`

4. 点击“开始”或按 `Enter`。
5. 进入分析页后等待进度步骤完成。
6. 分析结束后自动进入聊天页，先阅读首轮教学报告。

## 页面说明

### 输入页

输入页负责仓库地址录入和轻量格式校验。

- 输入过程中会触发格式校验。
- 失焦时会再次校验。
- 点击“开始”会正式创建会话并启动分析。
- 如果格式不正确，会在输入框下方看到红色提示。

### 分析页

分析页用于展示服务端推送的进度状态。

- 进度步骤完全由服务端状态驱动。
- 如果触发降级，会显示降级提示。
- 如果分析时间较长，会显示超时相关提示。
- 如果分析失败，会显示服务端返回的用户可读错误。

### 聊天页

聊天页包含三部分：

- 首轮报告或后续回答消息列表
- 建议问题按钮
- 追问输入框

当前前端行为：

- 首轮报告按 `initial_report_content` 的固定区块顺序渲染。
- 后续回答按 `structured_content` 的固定字段顺序渲染。
- 输入框在 `chatting + waiting_user` 之外会禁用。
- 输入框支持 `Enter` 发送，`Shift+Enter` 换行。
- 点击建议问题会直接发送该问题。

当前后端已经提供 LLM 工具上下文，但前端暂时还不会单独展示“本轮使用了哪些工具结果”。如果后续要改前端，优先考虑在 Agent 消息旁增加可展开的“参考来源/不确定项”区域，而不是改变现有聊天主流程。

## 推荐使用流程

推荐按下面顺序使用：

1. 先看“仓库概览”，建立项目类型和主语言印象。
2. 再看“先抓什么”和“当前仓库映射”，明确优先阅读对象。
3. 再看“入口候选”和“推荐第一步”，找到最可能的起点。
4. 最后根据“阅读路径预览”和“下一步建议”继续追问。

如果你刚接触一个仓库，优先问这些问题通常更有效：

- 入口候选之间有什么区别？
- 关键目录应该按什么顺序读？
- 主流程大致是怎么串起来的？
- 这个模块和入口是什么关系？

## 切换仓库

点击页面中的“切换仓库”按钮会：

- 清理当前活跃会话
- 关闭旧的 SSE 连接
- 回到输入页

由于第一版只有一个活跃会话，不建议同时在多个页面标签中混用不同仓库。

## 页面刷新与恢复

当前前端在页面加载时会调用 `GET /api/session`。

这意味着：

- 如果服务端仍保留当前会话，刷新后可以恢复到输入页、分析页或聊天页。
- 如果会话仍在分析或回答中，前端会按服务端快照重新连接相应 SSE。
- 如果会话已经被清理，页面会回到空闲输入态。

## 常见问题

### 1. 为什么提示路径不存在或不是目录？

请确认你输入的是仓库根目录的绝对路径，而不是文件路径或相对路径。

### 2. 为什么 GitHub URL 不可访问？

当前只支持公开 GitHub 仓库 URL。请确认仓库存在且可公开访问，并且本机 `git` 已在 PATH 中可用。

### 3. 为什么分析结果会出现降级提示？

常见原因包括：

- 仓库很大
- 主语言不是 Python
- 没找到可靠入口候选
- 证据不足以形成可靠流程或分层

这不是前端错误，而是系统在当前能力边界内给出的保守输出。

### 4. 为什么聊天输入框有时不能输入？

当服务端状态不是 `chatting + waiting_user` 时，前端会禁用输入框和建议按钮。例如 Agent 正在思考或正在流式输出时，输入会被暂时禁用。

### 5. 为什么命令行调接口时中文消息失败，但浏览器里正常？

在当前 Windows PowerShell + `curl.exe` 环境下，直接发送中文 JSON 可能受到 shell 编码或转义影响。浏览器 `fetch` 和 PowerShell `Invoke-RestMethod` 已完成联调验证，优先使用浏览器页面进行实际操作。

## 本地验证建议

在改动代码后，建议至少做这些检查：

### 后端

```bash
pytest -q -p no:cacheprovider
```

### 前端

```bash
cd frontend
npm run build
```

### 联调

至少手动走一遍：

1. 输入合法本地仓库路径
2. 等待分析完成
3. 发送一条追问
4. 点击一次建议问题
5. 切换仓库并确认回到输入页

## 排障建议

如果页面无法正常工作，优先按下面顺序检查：

1. 后端是否运行在 `127.0.0.1:8000`
2. 前端是否运行在 `127.0.0.1:5173`
3. 前端开发代理是否可访问 `http://127.0.0.1:5173/api/session`
4. 当前仓库输入是否为绝对路径或公开 GitHub URL
5. 浏览器开发者工具里是否存在接口 4xx/5xx 或 SSE 断流
