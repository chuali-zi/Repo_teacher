# PRD v4: Repo Tutor

版本：v4.0  
状态：目标需求明确版 / 架构设计输入版  
来源：基于 `docs/PRD_v1.txt` 的产品初心与 `docs/PRD_v3.md` 的范围收敛修订  
目标读者：产品、架构设计者、研发、测试  

## 0. v4 修订说明

### 0.1 本版目标

v4 的目标不是继续设计系统架构，而是把 Repo Tutor 第一版要达成的产品目标、用户体验、功能边界、输出要求和验收标准说清楚。

架构设计者读完本 PRD 后，应能清楚回答：

- 第一版产品到底要帮助用户完成什么。
- 第一版必须支持哪些输入、输出和对话行为。
- 哪些能力是 P0，哪些只是后续增强。
- 对静态分析结论、证据、置信度和不确定项有什么要求。
- 什么情况必须拒绝、降级或标注未知。
- 验收时如何判断产品是否达成目标。

本 PRD 不规定：

- 内部模块如何拆分。
- 数据结构字段如何设计。
- 具体类名、接口名、服务边界和依赖方向。
- 使用哪种索引、检索、上下文组装或缓存方案。
- 使用哪种框架、数据库、向量库或 Agent 编排方式。

这些内容由后续架构设计文档负责。

### 0.2 v3 的主要问题

v3 比 v1 更清楚地收敛了范围，但它已经明显进入架构设计层面，主要问题是：

- 把内部模块名称、模块职责和依赖方向写进了 PRD。
- 把输出数据契约写成接近接口设计的字段表。
- 把上下文管理、日志、缓存、分析器等实现思路提前固化。
- 把架构师交付要求写成架构方案清单，容易让 PRD 变成半份架构设计。

这些内容并非错误，但不应该在目标需求 PRD 中过度展开。v4 保留其产品边界和验收意图，删除或弱化实现设计。

### 0.3 v4 的取舍

v4 保留：

- 面向初学者的源码仓库教学定位。
- 第一版优先支持 Python 仓库。
- 只读、静态分析、候选结论、证据优先。
- 首轮教学报告、阅读路径、候选主流程、依赖来源解释、多轮追问。
- 不执行代码、不修改代码、不做通用 Agent OS 的边界。

v4 调整：

- 不再规定内部模块拆法。
- 不再规定上下文管理具体接口。
- 不再给出内部依赖图。
- 不再要求 PRD 中提前设计 RepositoryIndex、ContextPackage 等字段。
- 将“架构验收”改为“产品与行为验收”，只验证用户可见结果和必要边界。

## 1. 产品概述

### 1.1 产品名称

Repo Tutor

### 1.2 一句话定义

Repo Tutor 是一个面向初学者的只读源码仓库教学 Agent。它帮助用户从陌生仓库中识别入口、关键目录、主要模块、依赖来源、候选主流程和阅读顺序，并通过主动引导让用户逐步建立最小工程认知。

### 1.3 产品定位

Repo Tutor 是教学型 Agent，不是工程执行 Agent。

它要做的是：

- 教用户看懂一个真实代码仓库的基本结构。
- 帮用户知道先看哪里、为什么先看那里。
- 帮用户区分项目内部代码、标准库和第三方依赖。
- 帮用户沿着候选入口和候选主流程建立整体理解。
- 在多轮追问中维持当前仓库、目标和已讲解上下文。

它不做的是：

- 不运行仓库代码。
- 不安装依赖。
- 不运行测试。
- 不修改、删除或格式化仓库文件。
- 不生成 PR 或 commit。
- 不自动修复 bug。
- 不做完整代码质量审查。
- 不做通用 Agent OS。
- 不做多 Agent 编排平台。

### 1.4 产品本质

Repo Tutor 的第一层本质是教学产品：帮助缺少工程阅读经验的用户建立“看工程”的认知框架。

Repo Tutor 的第二层价值是后续实验载体：未来可以用于验证不同上下文管理策略。但第一版不把实验能力作为对外主卖点，也不要求 PRD 提前规定实验架构。

## 2. 用户与问题

### 2.1 目标用户

核心用户是学过基础编程、能看懂单个函数或类、但缺少真实工程阅读经验的初学者，包括：

- 大学生。
- 实习生。
- 初级开发者。
- 第一次阅读开源项目的人。

### 2.2 用户能力假设

用户通常能够：

- 提供一个本地仓库路径或 GitHub 公共仓库 URL。
- 看懂基础 Python 语法。
- 理解文件、目录、函数、类和 import 的基本含义。
- 跟随自然语言解释进行源码阅读。

用户通常不能稳定做到：

- 判断一个陌生仓库的入口在哪里。
- 判断哪些目录和模块最重要。
- 判断某个 import 是项目内部、标准库还是第三方库。
- 区分主路径和支撑路径。
- 沿着数据流、调用关系或模块关系阅读真实工程。
- 在多轮追问中自己维护当前仓库和当前学习目标。

### 2.3 核心痛点

初学者阅读真实工程时，最大困难通常不是某一行代码看不懂，而是：

- 不知道先看什么。
- 不知道哪些文件值得重点看。
- 不知道入口、模块、分层、依赖、主流程分别是什么意思。
- 不知道一个仓库是怎么“动起来”的。
- 看不出哪些代码是项目自己写的，哪些来自标准库或第三方库。
- 看了很多局部文件，但仍然无法拼出整体结构。

Repo Tutor 要解决的是：用户面对一个陌生仓库时，没有工程认知框架，不知道该看见什么、先理解什么，以及如何把模块和流程拼成一个整体。

## 3. 第一版目标与范围

### 3.1 第一版核心目标

第一版要完成一个“只读 Python 仓库教学闭环”。

用户输入一个支持范围内的仓库后，系统应能让用户知道：

- 这个仓库大概是什么类型的项目。
- 当前应该先观察哪些东西。
- 关键目录和主要模块在哪里。
- 入口候选在哪里。
- 应该按什么顺序阅读。
- 一条候选主流程大概如何经过入口、模块和依赖。
- 某些 import 来自项目内部、标准库、第三方库还是未知。
- 接下来应该继续看什么。

### 3.2 第一版必须支持

第一版 P0 支持：

- 本地仓库路径输入。
- GitHub 公共仓库 URL 输入。
- Python 仓库的只读分析与教学讲解。
- 文件树扫描和关键目录识别。
- Python 项目类型候选判断。
- Python 入口候选识别。
- Python import 来源识别。
- 仓库结构总览。
- 初学者可执行的阅读路径。
- 候选主流程讲解。
- 多轮追问。
- 用户显式切换学习目标。
- 证据引用、置信度和不确定项标注。
- 安全只读处理。

### 3.3 第一版有限支持

非 Python 仓库第一版只提供有限支持：

- 可以给出文件树级结构总览。
- 可以提示关键目录和关键文件。
- 必须明确标注“当前语言暂不完整支持”。

非 Python 仓库不得输出确定性的 Python 入口、Python 依赖来源或 Python 主流程结论。

### 3.4 第一版不支持

第一版明确不支持：

- 私有 GitHub 仓库认证。
- 多仓库联合分析。
- PDF、课件、设计文档混合输入。
- 在线运行代码。
- 安装依赖。
- 执行 shell 脚本。
- 运行测试。
- 修改仓库文件。
- 自动生成 PR 或 commit。
- 跨语言完整调用链分析。
- 完整动态运行时数据流分析。
- 复杂可视化架构图作为主交互。
- 高频互动式小测作为主流程。
- 通用 Agent OS。
- 多 Agent 编排平台。

## 4. 输入需求

### 4.1 仓库输入

系统应接受：

- 本地仓库路径。
- GitHub 公共仓库 URL。

系统应在输入无效时给出明确说明：

- 路径不存在。
- 路径不是可读取目录。
- GitHub URL 不合法。
- GitHub 仓库不可公开访问。
- 仓库过大导致第一轮无法完整分析。

### 4.2 用户追问输入

系统应支持用户围绕当前仓库继续提问，例如：

- “这个仓库先看哪里？”
- “入口在哪里？”
- “启动流程怎么走？”
- “登录流程怎么走？”
- “这个模块为什么存在？”
- “这个 import 是谁家的？”
- “只看数据库相关部分。”
- “先别讲整体，讲这个文件。”
- “讲浅一点。”
- “讲深一点。”

### 4.3 输入安全边界

系统只读取必要的仓库文本内容，不应读取敏感文件正文。

命中以下类型时，应默认不读取正文：

- `.env`
- `.env.*`
- `*.pem`
- `*.key`
- `*.crt`
- `id_rsa`
- `id_ed25519`
- `credentials*`
- `secrets*`
- `token*`

系统可以记录这些文件“存在但未读取内容”，用于说明安全处理。

## 5. 输出需求

### 5.1 首轮教学报告

首次分析成功后，系统必须输出首轮教学报告。

首轮报告必须让用户知道：

- 仓库大概是什么项目。
- 当前仓库最值得先抓住的观察框架是什么。
- 关键目录有哪些。
- 入口候选在哪里。
- 第一阶段最建议看什么。
- 为什么建议先看它。
- 后续可以继续问什么。
- 当前有哪些不确定项。

首轮报告不应只是文件树 dump，也不应一次性解释所有文件。

首轮默认信息量应适合初学者，建议控制在：

- 2 到 4 个核心认知点。
- 1 条推荐阅读起点。
- 1 组后续可选方向。
- 明确的不确定项说明。

### 5.2 仓库结构总览

系统应生成适合初学者理解的结构总览，说明：

- 仓库主要由哪些部分组成。
- 哪些目录或模块最重要。
- 每个关键目录大概承担什么角色。
- 哪些部分属于主路径，哪些更像支撑部分。
- 判断依据来自哪里。

结构总览必须服务于阅读理解，不应只是原样输出完整目录树。

### 5.3 阅读路径

系统应生成 3 到 6 步阅读路径。

每一步应说明：

- 看哪个文件、目录或模块。
- 为什么先看它。
- 看完应建立什么认知。
- 暂时可以跳过什么。
- 下一步看什么。
- 该建议基于哪些仓库证据。

阅读路径必须基于当前仓库，不得生成脱离仓库的通用学习路线。

### 5.4 入口候选解释

系统应识别并解释入口候选。

入口线索可以来自：

- `__main__.py`
- `main.py`
- `app.py`
- `manage.py`
- `if __name__ == "__main__"`
- `pyproject.toml` scripts
- `setup.py` entry_points
- README 运行命令。
- 常见框架约定入口，例如 Flask、FastAPI、Django。

输出应包含：

- 入口候选。
- 为什么认为它可能是入口。
- 置信度。
- 多入口时推荐先看哪个。
- 找不到入口时的替代阅读建议。

如果无法确认入口，系统必须输出“未知”或“未找到可靠入口候选”，不得硬猜。

### 5.5 依赖来源解释

系统应解释 import 来源，至少区分：

- 项目内部模块。
- Python 标准库。
- 第三方库或框架。
- 未知。

解释时应说明：

- import 名称。
- 它被哪些文件使用。
- 来源判断是什么。
- 判断依据是什么。
- 它在当前理解目标中是否值得展开。

系统不得把未知依赖强行归类为第三方依赖。

### 5.6 候选主流程讲解

系统应围绕一个入口候选讲解一条候选主流程。

第一版的最低要求是：

- 能说明入口文件或入口命令。
- 能说明入口附近直接调用或导入了哪些模块。
- 能给出模块级候选路径。
- 能说明每一步大概承担什么角色。
- 能标注当前流程讲解的可信程度。
- 能标注哪些部分无法确认。

第一版不要求：

- 证明真实运行时调用链。
- 覆盖动态注册、反射、装饰器副作用或框架注入。
- 完整追踪跨进程、跨服务、异步任务或外部系统的数据流。

如果证据不足，系统必须明确说明无法形成可靠流程，只给阅读建议。

### 5.7 模块关系讲解

系统应解释关键模块之间的关系。

对每个关键模块，应尽量说明：

- 它大概负责什么。
- 它和入口或主流程有什么关系。
- 它依赖或调用了哪些上游、下游或相邻模块。
- 它是否值得当前阶段深入阅读。
- 这些判断来自哪些文件、import、配置或 README 线索。

系统不得只根据目录名直接断定模块职责。

### 5.8 多轮回答

用户围绕同一仓库连续追问时，系统应保持：

- 当前仓库。
- 当前学习目标。
- 当前讲解阶段。
- 当前聚焦模块。
- 当前入口候选。
- 当前候选流程。
- 已讲解过的文件和模块。
- 上一轮下一步建议。

系统不应每一轮都重新从零介绍仓库。

### 5.9 学习目标切换

用户可以显式改变学习目标，例如：

- 只看结构。
- 只看入口。
- 只看启动流程。
- 只看某个模块。
- 只看依赖来源。
- 总结一下目前学到什么。
- 讲浅一点。
- 讲深一点。

系统必须围绕新目标重组下一轮回答。

例如：

- 用户说“只看依赖来源”时，下一轮不应继续展开阅读路径。
- 用户说“讲浅一点”时，应减少术语密度。
- 用户说“讲深一点”时，可以增加证据、调用关系和实现细节，但仍不能伪造确定性。

## 6. 证据、置信度与不确定项

### 6.1 证据要求

系统输出仓库相关判断时，应尽量引用证据。

可接受的证据包括：

- 文件路径。
- README 中的运行说明。
- 配置文件中的脚本、入口或依赖声明。
- import 关系。
- 函数、类或符号线索。
- 目录结构。
- 文件命名约定。

确定性判断必须有证据。没有证据时，应标注为推断或未知。

### 6.2 置信度要求

系统应对关键判断标注置信度。

置信度建议分为：

- 高：有直接配置、显式入口、README 命令或明确符号证据。
- 中：多条启发式证据一致，但缺少直接运行配置。
- 低：只有弱命名约定、目录结构或单一线索。
- 未知：无法可靠判断。

### 6.3 候选措辞要求

静态分析无法证明真实运行路径时，系统必须使用候选措辞，例如：

- 候选入口。
- 可能的入口。
- 候选主流程。
- 可能的模块关系。
- 基于当前证据推断。

系统不得把静态推断说成确定真实运行事实。

### 6.4 流程解释深度

候选主流程可以按解释深度分级：

- 无可靠流程：无法形成流程，只能给阅读建议。
- 入口附近流程：能识别入口及入口附近直接调用或导入。
- 模块级路径：能基于 import 和文件关系给出模块级候选路径。
- 符号级路径：能基于函数或类关系给出更细候选路径。

第一版 P0 至少应达到“入口附近流程”，并尽量在典型 Python 仓库上达到“模块级路径”。“符号级路径”不是第一版必须能力。

## 7. 教学体验要求

### 7.1 主动带路

系统默认应主动提出下一步，而不是等待用户自己知道该问什么。

每轮讲解结束时，应给出：

- 本轮重点。
- 本轮和整体的关系。
- 当前还不确定或还没讲的内容。
- 下一步建议。

### 7.2 先骨架后细节

默认教学顺序应是：

1. 仓库是什么。
2. 入口在哪里。
3. 先看哪些目录或模块。
4. 候选主流程怎么走。
5. 模块如何协作。
6. 局部实现细节。

用户显式改向时，可以打破该顺序。

### 7.3 单轮信息量可消化

系统不应一轮讲完所有内容。

默认每轮建议控制：

- 2 到 4 个核心认知点。
- 1 条主线。
- 1 个下一步建议。

### 7.4 连续讲解优先

系统不应在主流程里频繁插入“猜一猜”“你先回答”这类小任务。

如果需要自查问题，应放在阶段性总结中，并且默认不阻塞用户继续学习。

### 7.5 自然语言优先

第一版主要输出自然语言讲解。

图示、ASCII 图或复杂可视化只能作为辅助，不是主交互。

## 8. 功能需求

### F1. 仓库只读接入

优先级：P0

系统必须支持本地仓库路径和 GitHub 公共仓库 URL。

验收：

- 本地路径不存在时给出明确错误。
- GitHub 仓库不可访问时给出明确错误。
- 成功接入后不执行任何仓库代码。
- 成功接入后可以开始生成首轮教学报告或明确说明无法分析原因。

### F2. 文件树扫描与关键对象识别

优先级：P0

系统必须识别：

- 关键目录。
- 关键配置文件。
- README。
- Python 源码文件。
- 候选入口文件。
- 常见依赖声明位置。
- 需要忽略或不读取正文的文件。

验收：

- 输出关键目录摘要，而不是完整文件树 dump。
- 明确说明哪些文件因安全或体积原因未读取正文。
- 对仓库过大场景给出降级说明。

### F3. Python 项目画像

优先级：P0

系统应判断 Python 仓库的项目类型候选，例如：

- CLI 工具。
- Web 应用。
- Library / package。
- 脚本集合。
- Notebook 项目。
- 未知。

验收：

- 无法确认时输出未知。
- 非 Python 仓库不得强行输出 Python 项目画像。
- 判断应引用配置、README、文件结构或入口线索。

### F4. 入口候选识别

优先级：P0

系统必须识别入口候选，并给出理由、置信度和推荐阅读顺序。

验收：

- 每个入口候选都应有证据或明确说明是弱推断。
- 多入口时必须说明排序理由。
- 找不到入口时给出替代阅读路径。

### F5. import 来源识别

优先级：P0

系统必须解释 import 来源，至少支持内部模块、标准库、第三方库和未知四类。

验收：

- 至少能解释样例仓库中指定的代表性 import。
- 相对 import 优先按内部模块判断。
- 无法确认时输出未知。
- 不得把未知依赖强行归类。

### F6. 首轮教学报告

优先级：P0

系统必须在首次分析后生成首轮教学报告。

验收：

- 包含仓库一句话概览。
- 包含关键目录。
- 包含入口候选或未知说明。
- 包含推荐第一步。
- 包含后续可问方向。
- 包含不确定项。
- 输出面向初学者，而不是面向资深工程师。

### F7. 阅读路径生成

优先级：P0

系统必须生成 3 到 6 步阅读路径。

验收：

- 每一步都有目标、原因、学习收益、可暂时跳过内容和证据。
- 阅读路径来自当前仓库，而不是通用 Python 学习路径。

### F8. 候选主流程讲解

优先级：P0

系统必须能围绕一个入口候选讲解候选主流程。

验收：

- 能说明入口附近发生了什么。
- 能说明涉及哪些模块。
- 能说明数据或控制大概如何向后传递。
- 能标注置信度和不确定项。
- 证据不足时必须明确降级，不得硬编流程。

### F9. 模块关系讲解

优先级：P0

系统必须讲清关键模块的职责和关系。

验收：

- 至少覆盖首轮报告中提到的关键模块。
- 模块职责判断必须结合文件、import、README 或配置线索。
- 能说明该模块是否值得当前阶段深入。

### F10. 多轮上下文保持

优先级：P0

系统必须支持围绕同一仓库连续追问。

验收：

- 连续 5 轮追问中不丢失当前仓库。
- 连续 5 轮追问中不无故改变当前学习目标。
- 已解释内容不应反复从零讲。
- 用户指定模块后，后续回答应围绕该模块，直到用户改向或系统说明需要返回整体。

### F11. 学习目标切换

优先级：P0

系统必须支持用户显式切换学习目标。

验收：

- 用户说“只看依赖来源”时，下一轮围绕依赖来源回答。
- 用户说“只看启动流程”时，下一轮围绕启动流程回答。
- 用户说“讲浅一点”时，下一轮降低术语密度。
- 用户说“讲深一点”时，下一轮增加证据和实现细节，但仍标注不确定项。

### F12. 安全只读策略

优先级：P0

系统必须遵守只读和安全边界。

验收：

- 不执行仓库代码。
- 不运行测试。
- 不安装依赖。
- 不执行 shell 脚本。
- 不修改或删除文件。
- 不读取敏感文件正文。
- 不输出疑似密钥内容。

### F13. 阶段性总结

优先级：P1

系统在一个主题阶段结束时，应能总结：

- 用户已经理解了什么。
- 还缺什么关键认知。
- 下一阶段建议看什么。
- 可选自查问题。

自查问题不得阻塞继续学习。

### F14. 回答深浅控制

优先级：P1

系统应支持更稳定的深浅控制。

验收：

- “讲浅一点”时减少抽象术语和实现细节。
- “讲深一点”时增加证据、局部代码解释和调用关系。
- 深浅变化不改变事实边界。

### F15. 后续实验预留

优先级：P2

第一版可以为未来上下文管理实验保留必要信息，但本 PRD 不规定实验实现方式。

产品层面只要求：

- 后续能追踪一次回答大概用了哪些证据。
- 后续能评估多轮回答是否保持上下文。
- 不因实验预留影响第一版用户体验。

## 9. 页面与交互要求

### 9.1 第一版交互形态

第一版建议采用 CLI 或极简对话式界面。

PRD 不要求重 UI。

### 9.2 基础输入

界面至少应支持：

- 输入本地仓库路径。
- 输入 GitHub 公共仓库 URL。
- 输入自然语言追问。

### 9.3 基础状态展示

界面应能让用户知道：

- 当前仓库是什么。
- 当前学习目标是什么。
- 当前聚焦模块或流程是什么。
- 下一步建议是什么。

状态展示可以是文本形式，不要求复杂 UI。

### 9.4 每轮输出结构

每轮回答建议包含：

- 本轮重点。
- 直接解释。
- 与整体的关系。
- 证据或判断依据。
- 不确定项。
- 下一步建议。

## 10. 非功能需求

### 10.1 准确性

- 确定性结论必须有证据。
- 推断必须标注为推断。
- 无法判断必须输出未知。
- 候选流程必须标注候选性质和置信度。

### 10.2 可理解性

- 默认语言面向初学者。
- 首轮不应堆砌术语。
- 解释应围绕“为什么先看这里”和“它在整体里做什么”展开。
- 代码细节应服务于工程理解，而不是逐行翻译。

### 10.3 性能体验

目标体验：

- 小型仓库：少于 500 个源码文件，30 秒内开始首轮输出或给出进度说明。
- 中型仓库：500 到 3000 个源码文件，90 秒内开始首轮输出或给出阶段性结果。
- 大型仓库：超过 3000 个源码文件，必须告知用户已启用降级分析，并优先输出可用的结构总览和阅读起点。

### 10.4 安全与隐私

- 不读取敏感文件正文。
- 不输出疑似密钥。
- 不执行仓库代码。
- GitHub 第一版只支持公开仓库。
- 本地仓库访问必须防止越界读取。

### 10.5 可测试性

系统行为必须能通过固定样例仓库验收。

至少应能验证：

- 首轮报告是否包含必需内容。
- 入口候选是否正确或合理标注未知。
- 阅读路径是否来自仓库证据。
- import 来源解释是否正确。
- 候选主流程是否标注置信度和不确定项。
- 多轮上下文是否保持。
- 安全文件是否未读取正文。

## 11. 验收标准

### 11.1 固定样例仓库

第一版至少准备 3 个固定测试仓库，使用固定 commit 或本地 fixture：

1. 小型 Python CLI 项目。
2. 小型 Python Web 项目，例如 Flask、FastAPI 或 Django。
3. Python library / package 项目。

每个样例仓库必须预先标注：

- 期望入口候选或允许未知的理由。
- 关键目录。
- 主要内部模块。
- 代表性标准库 import。
- 代表性第三方 import。
- 至少一条候选主流程或无法形成流程的理由。

### 11.2 首轮报告验收

每个样例仓库的首轮报告必须包含：

- 仓库一句话概览。
- 主语言和项目类型候选。
- 关键目录。
- 入口候选或未知说明。
- 推荐第一步。
- 3 到 6 步阅读路径或阅读路径预览。
- 下一步建议。
- 不确定项。

### 11.3 功能验收

每个样例仓库必须通过：

- 成功接入仓库。
- 成功生成教学可用的结构总览。
- 至少识别 1 个入口候选，或明确输出未知并给出替代路径。
- 能解释至少 5 个 import 来源。
- 能输出 1 条候选主流程，或明确说明无法形成流程。
- 支持连续 5 轮追问。
- 支持用户切换学习目标。

### 11.4 证据验收

每个确定性判断必须关联至少 1 条证据。

以下判断必须有证据或明确标注未知：

- 项目类型。
- 入口候选。
- 关键目录。
- 主要模块职责。
- import 来源。
- 候选主流程步骤。

### 11.5 多轮状态验收

连续 5 轮对话中：

- 当前仓库不应丢失。
- 当前学习目标不应无故变化。
- 当前聚焦模块变化时必须符合用户意图或系统解释。
- 已解释内容不应重复从零讲。
- 下一步建议应随对话推进而变化。

### 11.6 学习目标切换验收

给定同一个仓库：

- 用户切换到“只看入口”后，系统应聚焦入口候选。
- 用户切换到“只看依赖来源”后，系统应聚焦 import 来源。
- 用户切换到“只看某个模块”后，系统应聚焦该模块。
- 用户要求“讲浅一点”后，回答应更少术语。
- 用户要求“讲深一点”后，回答可更细，但不得伪造确定性。

### 11.7 安全验收

测试仓库中放置 `.env`、`secrets.json`、`id_rsa` 等文件时：

- 系统不得读取正文。
- 输出不得包含敏感文件内容。
- 只能说明这些文件因安全策略未读取。

### 11.8 非 Python 仓库验收

给定一个非 Python 仓库：

- 系统可以输出文件树级结构总览。
- 系统必须明确说明当前语言暂不完整支持。
- 系统不得输出确定性的 Python 入口。
- 系统不得输出确定性的 Python import 来源。
- 系统不得输出伪造的 Python 主流程。

## 12. 优先级

### 12.1 P0：第一版必须完成

- 本地仓库路径接入。
- GitHub 公共仓库 URL 接入。
- 安全只读访问。
- 文件树扫描和关键目录识别。
- Python 项目画像。
- 入口候选识别。
- import 来源识别。
- 首轮教学报告。
- 仓库结构总览。
- 3 到 6 步阅读路径。
- 候选主流程讲解。
- 模块关系讲解。
- 证据、置信度、不确定项标注。
- 多轮追问上下文保持。
- 学习目标切换。
- 非 Python 仓库降级支持。

### 12.2 P1：强烈建议完成

- 阶段性总结。
- 更稳定的深浅讲解控制。
- 更细的模块职责解释。
- 更好的候选流程说明。
- 对大仓库更好的阶段性输出。

### 12.3 P2：后续版本考虑

- 更完整的符号级流程分析。
- 更多语言支持。
- 私有仓库认证。
- 文档混合输入。
- 更复杂可视化。
- 后续上下文管理实验。
- 多种上下文策略对比。

## 13. 风险与约束

### 13.1 静态分析天然不完整

系统不能假装知道真实运行时行为。对于动态注册、框架注入、反射、装饰器副作用、配置驱动流程等情况，必须标注不确定。

### 13.2 P0 范围容易膨胀

入口识别、依赖来源、阅读路径和候选流程都可以无限做深。第一版只要求达到初学者教学可用，不要求完整静态分析器能力。

### 13.3 教学输出不能变成审计报告

系统不是代码审查工具，不应把重点放在 bug、坏味道、性能问题或重构建议上。除非用户明确询问，否则默认围绕理解仓库展开。

### 13.4 后续实验不能压过产品闭环

上下文管理实验是后续价值，但第一版成功标准是用户是否能更好地理解仓库，而不是实验框架是否完整。

## 14. 架构设计输入说明

本 PRD 给架构设计者的输入是“目标和边界”，不是“架构方案”。

架构设计者应基于本 PRD 自行设计：

- 如何接入仓库。
- 如何扫描和索引文件。
- 如何做 Python 静态分析。
- 如何组织证据。
- 如何维护多轮上下文。
- 如何生成教学回答。
- 如何做安全隔离。
- 如何设计测试和日志。

架构方案不得违背以下产品边界：

- 对外是单 Agent 教学体验。
- 第一版只读，不执行，不修改。
- 第一版优先支持 Python 仓库。
- 所有确定性结论都需要证据。
- 流程讲解必须标注候选性质。
- 多轮追问必须保持当前仓库和学习目标。
- 不扩展为通用 Agent OS、代码执行平台、自动修复工具或多 Agent 编排平台。

## 15. 给架构设计者的一句话任务

请基于本 PRD 设计 Repo Tutor 第一版架构：它应是一个面向初学者的单 Agent 教学系统，第一版只做 Python 仓库的只读理解与教学引导，必须交付结构总览、阅读路径、入口候选、依赖来源解释、候选主流程、多轮上下文保持和学习目标切换，并严格避免扩展成代码执行、自动修复、通用 Agent OS 或多 Agent 平台。

## 16. 产品成功标准

如果第一版成功，应满足下面这句话：

一个会基础 Python 但不会阅读真实工程的用户，在使用 Repo Tutor 后，能够说清一个陌生仓库的项目类型、入口候选、关键目录、主要模块、依赖来源、候选主流程和下一步阅读顺序，并能在 Agent 主动引导下持续深入，而不是在文件堆里迷失。
