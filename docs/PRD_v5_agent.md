## <a id="product"></a>PRODUCT — 产品定义与教学主线

### 产品名称
`Repo Tutor`

### 一句话定义
面向初学者的**只读**源码仓库教学 Agent。帮助用户在陌生仓库中建立最小工程认知，识别入口、模块、分层、依赖来源、候选数据流/主流程和阅读顺序，并通过主动引导持续深入。

### 产品类型
- **教学型 Agent**，不是工程执行 Agent。
- **单 Agent 体验**，不是多 Agent 编排平台。
- **第一版只读**，不执行、不修改仓库。

### 产品本质
Repo Tutor 有两层定位：

1. **教学产品壳子**
   - 帮用户理解陌生代码仓库
   - 帮用户建立工程认知
   - 帮用户沿主流程与数据流逐步深入
2. **后续实验载体**
   - 为未来上下文管理实验预留接口
   - 但第一版不把实验能力作为对外主卖点

### 第一版教学主线
第一版不是“你问我答”的仓库问答器，而是围绕以下主线组织教学：

1. 先建立观察框架：入口、模块、分层、依赖、主流程分别是什么
2. 再映射到当前仓库：这些东西在这个仓库里分别体现在哪里
3. 再给阅读起点：先看哪里，为什么先看这里
4. 再沿候选数据流/主流程展开：系统大概怎么动起来
5. 再补充模块关系与局部实现细节

### 产品成功标准（一句话）
一个会基础编程但不会阅读真实工程的用户，在使用 Repo Tutor 后，能够说清一个陌生仓库的入口候选、关键目录、核心模块、教学式分层、依赖来源、候选数据流/主流程和阅读顺序，并能在 Agent 主动带路下持续深入，而不是在文件堆里迷失。

## <a id="users"></a>USERS — 用户与能力假设

### 目标用户
大学生、实习生、初级开发者、第一次阅读开源项目的人。

### 用户能做的事（输入前提）
| 能力 | 说明 |
|------|------|
| 提供仓库路径或 URL | 本地路径或 GitHub 公共仓库 URL |
| 看懂基础 Python 语法 | 单个函数/类可读 |
| 理解文件、目录、函数、类、import | 基础概念已知 |
| 跟随自然语言解释阅读源码 | 无需自行规划阅读顺序 |

### 用户不能稳定做到的事（Agent 必须补足的能力）
- 判断陌生仓库的入口在哪里
- 判断哪些目录/模块最重要
- 判断 import 来源（内部/标准库/第三方/未知）
- 区分主路径和支撑路径
- 建立入口层、业务层、数据层、工具层等工程分层直觉
- 沿数据流、调用关系阅读真实工程
- 在多轮追问中自己维护当前仓库和学习目标

## <a id="scope"></a>SCOPE — 范围与优先级

### P0：第一版必须完成
```
- 本地仓库路径接入
- GitHub 公共仓库 URL 接入
- 安全只读访问
- 文件树扫描和关键目录识别
- Python 项目画像（类型候选）
- 入口候选识别
- import 来源识别（内部/标准库/第三方/未知）
- 首轮教学报告
- 仓库结构总览
- 3–6 步阅读路径
- 候选数据流/主流程骨架讲解
- 模块关系讲解
- 教学式分层视图
- 证据/置信度/不确定项标注
- 多轮追问上下文保持（≥5 轮）
- 学习目标切换
- 非 Python 仓库降级支持
```

### P1：强烈建议完成
```
- 阶段性总结
- 更稳定的深浅讲解控制
- 更细的模块职责解释
- 更好的候选流程说明
- 更稳定的大仓库阶段性输出
```

### P2：后续版本考虑
```
- 符号级流程分析
- 更多语言支持
- 私有仓库认证
- 文档混合输入
- 更复杂可视化
- 更完整的上下文管理实验
```

### 明确不支持（第一版）
```
- 私有 GitHub 仓库认证
- 多仓库联合分析
- PDF/课件/设计文档混合输入
- 在线运行代码
- 安装依赖
- 执行 shell 脚本
- 运行测试
- 修改仓库文件
- 自动生成 PR 或 commit
- 跨语言完整调用链分析
- 完整动态运行时数据流分析
- 复杂可视化架构图作为主交互
- 高频互动式小测作为主流程
- 通用 Agent OS
- 多 Agent 编排平台
```

## <a id="input"></a>INPUT — 输入规范

### 仓库输入类型
| 类型 | 格式 | 备注 |
|------|------|------|
| 本地仓库路径 | 绝对路径字符串 | 必须是可读目录 |
| GitHub 公共仓库 URL | `https://github.com/owner/repo` | 必须可公开访问 |

### 输入错误处理（必须明确提示）
| 错误情况 | 必须输出 |
|----------|----------|
| 路径不存在 | 明确错误说明 |
| 路径不是可读取目录 | 明确错误说明 |
| GitHub URL 不合法 | 明确错误说明 |
| GitHub 仓库不可公开访问 | 明确错误说明 |
| 仓库过大导致首轮无法完整分析 | 明确说明，启用降级分析 |

### 用户追问输入（多轮，围绕当前仓库）
Agent 必须支持但不限于以下追问形式：
```
"这个仓库先看哪里？"
"入口在哪里？"
"启动流程怎么走？"
"登录流程怎么走？"
"这个模块为什么存在？"
"这个 import 是谁家的？"
"这个仓库大体怎么分层？"
"先别讲模块，先讲数据流。"
"只看数据库相关部分。"
"讲浅一点。"
"讲深一点。"
```

### 输入安全边界 — 敏感文件黑名单
下列文件命中时，**默认不读取正文**，只记录“文件存在但未读取内容”：
```
.env
.env.*
*.pem
*.key
*.crt
id_rsa
id_ed25519
credentials*
secrets*
token*
```

## <a id="output"></a>OUTPUT — 输出规范

### OUT-1: 首轮教学报告 `[P0]`

**触发条件**：首次仓库接入分析成功后必须输出。

**必须包含的内容**：
| 字段 | 说明 |
|------|------|
| 仓库一句话概览 | 是什么项目 |
| 当前看仓库先抓什么 | 入口/模块/分层/依赖/主流程中的 2–4 个重点 |
| 当前仓库里的概念映射 | 上述重点在当前仓库里的落点 |
| 主语言 + 项目类型候选 | 附置信度 |
| 关键目录 | 列举 + 说明角色 |
| 入口候选 | 或明确输出“未知”并给替代路径 |
| 推荐第一步 | 先看什么，为什么 |
| 3–6 步阅读路径预览 | 或完整阅读路径 |
| 下一步建议 | 用户可继续问的方向 |
| 不确定项 | 当前无法确认的内容 |

**禁止**：
- 不得只是文件树 dump
- 不得一次性解释所有文件
- 不得缺失“观察框架”和“当前仓库映射”

---

### OUT-2: 仓库结构总览 `[P0]`

**内容要求**：
- 仓库主要由哪些部分组成
- 哪些目录/模块最重要
- 每个关键目录大概承担什么角色
- 哪些是主路径，哪些是支撑部分
- 判断依据来自哪里

**禁止**：不得只是原样输出完整目录树。

---

### OUT-3: 阅读路径 `[P0]`

**规格**：生成 3–6 步阅读路径。

**每一步必须包含**：
| 字段 | 说明 |
|------|------|
| 目标 | 看哪个文件/目录/模块 |
| 原因 | 为什么先看它 |
| 学习收益 | 看完应建立什么认知 |
| 可跳过内容 | 暂时可以跳过什么 |
| 下一步 | 下一步看什么 |
| 证据 | 该建议基于哪些仓库证据 |

**约束**：必须基于当前仓库，不得生成通用 Python 学习路线。

---

### OUT-4: 入口候选解释 `[P0]`

**入口线索来源**（Agent 应检查）：
```
__main__.py
main.py
app.py
manage.py
if __name__ == "__main__"
pyproject.toml scripts 字段
setup.py entry_points 字段
README 运行命令
框架约定入口（Flask, FastAPI, Django）
```

**每个入口候选必须包含**：
| 字段 | 说明 |
|------|------|
| 候选文件/命令 | 具体位置 |
| 理由 | 为什么认为是入口 |
| 置信度 | 高/中/低/未知 |
| 多入口时的排序理由 | 推荐先看哪个 |

**找不到入口时**：必须输出“未知”或“未找到可靠入口候选”，并给出替代阅读建议。**不得硬猜。**

---

### OUT-5: 依赖来源解释 `[P0]`

**分类**（必须支持）：
| 来源类型 | 说明 |
|----------|------|
| 项目内部模块 | 仓库内的自有代码 |
| Python 标准库 | stdlib，无需安装 |
| 第三方库/框架 | pip 安装的外部依赖 |
| 未知 | 无法可靠判断 |

**每条解释必须包含**：
- import 名称
- 被哪些文件使用
- 来源判断结论
- 判断依据
- 是否值得当前阶段展开

**约束**：不得把未知依赖强行归类为第三方依赖。

---

### OUT-6: 候选数据流/主流程骨架讲解 `[P0]`

**第一版目标**：不是证明真实运行时调用链，而是形成可教学的流程骨架。

**流程解释深度分级**：

| 深度级别 | 描述 | v1/v5 要求 |
|----------|------|------------|
| 无可靠流程 | 无法形成流程，只给阅读建议 | 必须支持此降级 |
| 入口附近流程 | 识别入口及入口附近直接调用/导入 | P0 最低要求 |
| 模块级路径 | 基于 import、配置、README、文件关系给出候选路径 | P0 在典型 Python 仓库上应达到 |
| 教学式数据流骨架 | 说明输入从哪里进入、经过哪些层、结果去向何处 | **P0 核心目标** |
| 符号级路径 | 基于函数/类关系给出更细候选路径 | P2 |

**每条流程讲解必须包含**：
- 入口文件或入口命令
- 输入从哪里进入系统，或明确未知
- 入口附近直接调用/导入了哪些模块
- 候选模块级路径
- 数据或控制大概经过哪些层
- 结果大概去向哪里，或明确未知
- 当前流程的可信程度标注
- 无法确认的部分标注

**明确不要求**：
- 证明真实运行时调用链
- 覆盖动态注册、反射、装饰器副作用、框架注入
- 完整追踪跨进程、跨服务、异步任务、外部系统数据流

**证据不足时**：必须明确说明无法形成可靠流程，只给阅读建议。不得伪造流程闭环。

---

### OUT-7: 教学式分层视图 `[P0]`

**目的**：帮助初学者建立工程感，而不是追求架构学术严谨。

**允许的层次示例**：
```
入口层
路由/控制层
业务逻辑层
数据访问层
工具/配置层
```

**输出必须包含**：
- 当前仓库是否能形成教学式分层视图
- 每一层对应的目录/模块候选
- 每一层在整体中的角色
- 哪些层属于主路径，哪些更偏支撑
- 判断依据
- 无法稳定分层时明确标注“启发式分层”或“当前未知”

**约束**：
- 不得仅凭目录名强行断言层次
- 不要求绝对正确，但必须服务于教学理解

---

### OUT-8: 模块关系讲解 `[P0]`

**每个关键模块应说明**：
- 大概负责什么
- 与入口或候选主流程的关系
- 位于哪一层或接近哪一层
- 依赖/调用了哪些上下游模块
- 是否值得当前阶段深入阅读
- 判断来自哪些文件、import、配置、README 线索

**约束**：不得只根据目录名直接断定模块职责。

---

### OUT-9: 多轮回答 `[P0]`

**Agent 跨轮必须保持的状态**：
```
当前仓库
当前学习目标
当前讲解阶段
当前聚焦模块
当前入口候选
当前候选流程
当前分层理解结果
已讲解过的文件和模块
上一轮下一步建议
```

**约束**：不应每一轮都重新从零介绍仓库。

---

### OUT-10: 学习目标切换响应 `[P0]`

**用户可显式切换的目标及 Agent 对应行为**：
| 用户指令 | Agent 行为 |
|----------|-----------|
| 只看结构 | 聚焦仓库结构总览 |
| 只看入口 | 聚焦入口候选 |
| 只看启动流程 | 聚焦候选数据流/主流程 |
| 只看某个模块 | 聚焦该模块 |
| 只看依赖来源 | 聚焦 import 来源，不展开阅读路径 |
| 只看分层 | 聚焦教学式分层视图 |
| 总结一下目前学到什么 | 生成阶段性总结 |
| 讲浅一点 | 减少术语密度 |
| 讲深一点 | 增加证据和实现细节，但仍标注不确定项 |

---

### OUT-11: 每轮回答结构建议 `[P0]`

每轮回答建议包含以下结构：
```
1. 本轮重点
2. 直接解释
3. 与整体的关系
4. 证据或判断依据
5. 不确定项
6. 下一步建议
```

---

### OUT-12: 非 Python 仓库降级输出 `[P0]`

**允许**：
- 文件树级结构总览
- 关键目录和关键文件提示
- 保守的阅读起点建议

**必须**：明确标注“当前语言暂不完整支持”。

**禁止**：
- 不得输出确定性的 Python 入口
- 不得输出确定性的 Python 依赖来源
- 不得输出伪造的 Python 主流程
- 不得伪造教学式分层视图

## <a id="analysis"></a>ANALYSIS — 分析模块必须产出的教学骨架

### 目标
分析模块不是只给“仓库事实列表”，而是要给上层回答组织提供**教学可用骨架**。

### 分析模块最低产出
分析模块至少要能产出以下内容，即使部分字段为“未知”也必须显式输出：

| 产物 | 作用 |
|------|------|
| 仓库概览候选 | 帮助解释“这是什么项目” |
| 关键目录/模块候选 | 帮助解释“哪些东西重要” |
| 入口候选集合 | 帮助解释“系统从哪里开始” |
| 依赖来源候选 | 帮助解释“谁是自己人，谁是外部依赖” |
| 教学式分层候选 | 帮助解释“这些模块在工程里怎么分工” |
| 候选数据流/主流程骨架 | 帮助解释“系统大概怎么动起来” |
| 阅读起点建议 | 帮助解释“先看哪里” |
| 证据集合 | 支撑所有确定性或候选性结论 |
| 未知项集合 | 防止硬编结论 |

### 分析模块必须遵守的组织原则
1. **先教学，再分析细节**
   - 分析结果必须支持“观察框架 -> 当前仓库映射 -> 推荐阅读起点”的首轮讲解顺序。
2. **流程优先于零散事实**
   - 不能只有入口、目录、import 的孤立识别结果。
   - 必须尽量组织为“入口 -> 层 -> 模块 -> 去向”的流程骨架。
3. **分层是教学视图，不是学术建模**
   - 可以启发式，但不能完全缺席。
4. **未知优先于硬猜**
   - 证据不足时输出未知、候选或替代阅读路径。
5. **结果必须可被上层组装**
   - 分析结果不能硬编码进单一 prompt 文案。
   - 每个结论应可被回答层按主题重组，如结构、入口、流程、分层、依赖。

### 对架构师的直接约束
架构设计如果让分析模块只能稳定产出“目录摘要 + 入口启发式 + import 分类”，则**不满足本 PRD**。

分析模块至少要支持上层稳定回答以下问题：
```
这个仓库先看什么？
入口在哪里，为什么它像入口？
这个仓库大体怎么分层？
系统大概怎么动起来？
数据或控制大概经过哪些模块？
这个 import 是内部还是外部？
为什么建议先看这个模块？
```

## <a id="evidence"></a>EVIDENCE — 证据、置信度与候选措辞

### 可接受的证据类型
```
- 文件路径
- README 中的运行说明
- 配置文件中的脚本、入口、依赖声明
- import 关系
- 函数、类、符号线索
- 目录结构
- 文件命名约定
```

### 置信度分级
| 级别 | 条件 |
|------|------|
| 高 | 有直接配置、显式入口、README 命令或明确符号证据 |
| 中 | 多条启发式证据一致，但缺少直接运行配置 |
| 低 | 只有弱命名约定、目录结构或单一线索 |
| 未知 | 无法可靠判断 |

### 候选措辞要求
静态分析无法证明真实运行路径时，**必须**使用候选措辞：
```
候选入口
可能的入口
候选主流程
候选数据流
可能的模块关系
启发式分层
基于当前证据推断
```

**禁止**：不得把静态推断说成确定真实运行事实。

### 证据约束
- 确定性判断必须有证据
- 没有证据时标注为“推断”或“未知”
- 每个确定性判断必须关联至少 1 条证据

## <a id="ux"></a>UX — 教学体验规则

### 规则 1：主动带路
每轮讲解结束时，Agent 必须给出：
- 本轮重点
- 本轮和整体的关系
- 当前还不确定或还没讲的内容
- 下一步建议

### 规则 2：先骨架后细节
默认教学顺序：
```
1. 这个仓库是什么
2. 先抓入口、模块、分层、依赖、主流程中的关键点
3. 这些关键点在当前仓库里的映射
4. 先看哪些目录或模块
5. 候选数据流/主流程怎么走
6. 模块如何协作
7. 局部实现细节
```
用户显式改向时，可以打破该顺序。

### 规则 3：单轮信息量可消化
默认每轮控制：
- 2–4 个核心认知点
- 1 条主线
- 1 个下一步建议

### 规则 4：连续讲解优先
- 不应在主流程里频繁插入“猜一猜”“你先回答”等小任务
- 自查问题只在阶段性总结中出现，且默认不阻塞用户继续学习

### 规则 5：自然语言优先
- 第一版主要输出自然语言讲解
- 图示或 ASCII 图只能作为辅助，不是主交互

- [ ] 首轮回答明确说明这些概念在当前仓库里的落点

## <a id="nfr"></a>NFR — 非功能需求

### 准确性
| 规则 | 描述 |
|------|------|
| 证据强制 | 确定性结论必须有证据 |
| 推断标注 | 推断必须标注为推断 |
| 未知输出 | 无法判断时必须输出“未知” |
| 候选标注 | 候选流程必须标注候选性质和置信度 |

### 可理解性
- 默认语言面向初学者
- 首轮不堆砌术语
- 解释围绕“为什么先看这里”和“它在整体里做什么”
- 代码细节服务于工程理解，不逐行翻译

### 性能目标
| 仓库规模 | 源码文件数 | 目标 |
|----------|-----------|------|
| 小型 | <500 | 30 秒内开始首轮输出或给进度说明 |
| 中型 | 500–3000 | 90 秒内开始首轮输出或给阶段性结果 |
| 大型 | >3000 | 必须告知启用降级分析，优先输出结构总览和阅读起点 |

### 安全与隐私
```
- 不读取敏感文件正文
- 不输出疑似密钥
- 不执行仓库代码
- GitHub 第一版只支持公开仓库
- 本地仓库访问必须防止越界读取
```

### 可测试性
系统行为必须能通过固定样例仓库验收，至少能验证：
```
- 首轮报告是否包含必需内容
- 工程认知引导是否存在
- 入口候选是否正确或合理标注未知
- 阅读路径是否来自仓库证据
- import 来源解释是否正确
- 候选数据流/主流程是否标注置信度和不确定项
- 教学式分层视图是否存在或合理标注未知
- 多轮上下文是否保持
- 安全文件是否未读取正文
```

## <a id="acceptance"></a>ACCEPTANCE — 验收标准

### 样例仓库要求
第一版至少准备 3 个固定测试仓库（使用固定 commit 或本地 fixture）：

| 编号 | 类型 |
|------|------|
| T1 | 小型 Python CLI 项目 |
| T2 | 小型 Python Web 项目（Flask/FastAPI/Django） |
| T3 | Python library/package 项目 |

每个样例仓库必须预先标注：
- 期望入口候选或允许未知的理由
- 关键目录
- 主要内部模块
- 代表性标准库 import（≥3 个）
- 代表性第三方 import（≥3 个）
- 至少 1 条候选主流程或候选数据流骨架，或无法形成流程的理由
- 一个可接受的教学式分层视图，或无法稳定分层的理由

### 首轮报告验收（每个样例仓库）
- [ ] 仓库一句话概览
- [ ] 当前先抓什么
- [ ] 当前仓库里的概念映射
- [ ] 主语言和项目类型候选
- [ ] 关键目录
- [ ] 入口候选或未知说明
- [ ] 推荐第一步
- [ ] 3–6 步阅读路径或阅读路径预览
- [ ] 下一步建议
- [ ] 不确定项

### 功能验收（每个样例仓库）
- [ ] 成功接入仓库
- [ ] 成功生成教学可用的结构总览
- [ ] 至少识别 1 个入口候选，或明确输出未知并给出替代路径
- [ ] 能解释至少 5 个 import 来源
- [ ] 能输出 1 条候选数据流/主流程骨架，或明确说明无法形成流程
- [ ] 能输出教学式分层视图，或明确说明当前无法稳定分层
- [ ] 支持连续 5 轮追问
- [ ] 支持用户切换学习目标

### 证据验收
以下判断必须有证据或明确标注未知：
- [ ] 项目类型
- [ ] 入口候选
- [ ] 关键目录
- [ ] 主要模块职责
- [ ] import 来源
- [ ] 候选主流程步骤
- [ ] 分层视图映射

### 多轮状态验收（连续 5 轮对话中）
- [ ] 当前仓库不丢失
- [ ] 当前学习目标不无故变化
- [ ] 当前聚焦模块变化时符合用户意图或有系统解释
- [ ] 当前分层理解结果可延续使用
- [ ] 已解释内容不重复从零讲
- [ ] 下一步建议随对话推进而变化

### 学习目标切换验收
- [ ] 切换到“只看入口”后，系统聚焦入口候选
- [ ] 切换到“只看依赖来源”后，系统聚焦 import 来源
- [ ] 切换到“只看分层”后，系统聚焦教学式分层视图
- [ ] 切换到“只看某个模块”后，系统聚焦该模块
- [ ] 要求“讲浅一点”后，回答更少术语
- [ ] 要求“讲深一点”后，回答可更细，但不伪造确定性

### 安全验收
测试仓库中放置 `.env`、`secrets.json`、`id_rsa` 等文件时：
- [ ] 系统不读取正文
- [ ] 输出不包含敏感文件内容
- [ ] 只说明这些文件因安全策略未读取

### 非 Python 仓库验收
- [ ] 可以输出文件树级结构总览
- [ ] 必须明确说明当前语言暂不完整支持
- [ ] 不输出确定性的 Python 入口
- [ ] 不输出确定性的 Python import 来源
- [ ] 不输出伪造的 Python 主流程
- [ ] 不伪造教学式分层视图

## <a id="constraints"></a>CONSTRAINTS — 硬性约束

架构方案与分析模块不得违背以下产品边界：

```
1. 对外是单 Agent 教学体验
2. 第一版只读，不执行，不修改
3. 第一版优先支持 Python 仓库
4. 所有确定性结论都需要证据
5. 流程讲解必须标注候选性质
6. 多轮追问必须保持当前仓库和学习目标
7. 必须支持工程认知引导，而不是只输出分析结果列表
8. 必须支持教学式分层视图，而不是只输出目录摘要
9. 必须尽量形成候选数据流/主流程骨架，而不是只识别入口附近代码
10. 不扩展为通用 Agent OS、代码执行平台、自动修复工具或多 Agent 编排平台
```
