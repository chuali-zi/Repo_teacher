# DeepResearch Long Report Solution

> 日期：2026-04-20  
> 目标：给出能满足“中文、全面、教学型、至少 3000 字、强调架构与设计动机”的 deepresearch 解决方案  
> 设计前提：`quick_guide` 的交互与保守原则不变；`deep_research` 作为独立模式升级，不影响原本的对话体验与 SSE 基本契约。

## 1. 需求重述

新 deepresearch 的目标不是“更长一点的首轮摘要”，而是“成型的中文教学长报告”。

必须满足：

- 简体中文输出，英语只允许出现在代码标识符、类名、文件名、协议名里
- 正文不少于 3000 个中文汉字
- 不得沦为路径清单、文件摘要拼接、逐句机械引用
- 必须回答这些问题：
  - 项目整体架构是什么
  - 为什么这样架构
  - 这样架构有什么收益与代价
  - 项目用了哪些核心数据结构，为什么
  - 模块如何连接、如何协作
  - 哪些地方做了解耦，为什么要解耦
  - 每个核心模块的职责和实现是什么
  - 新人应该按什么路径去读
  - 哪些判断已验证，哪些仍是推断
- 不改变原版交互：
  - 仍然从 `web/` 现有模式切换进入
  - 仍然经由 `/api/repo` 和 `/api/analysis/stream`
  - 仍然把正文渲染在 `MessageDto.raw_text`

## 2. 三种可行路线

## 2.1 方案 A：继续增强当前确定性模板

思路：

- 保留 `backend/deep_research/pipeline.py` 的确定性路线
- 增加更多 AST 分析、更多章节模板、更多中文文案

优点：

- 改动小
- 风险低
- 不会碰现有 LLM tool loop

缺点：

- 很难写出真正的教学性长报告
- “为什么这样设计”这类问题无法仅靠 AST 模板回答
- 会继续产生很强的模板味和拼接味
- 越补越像规则引擎，维护成本会持续上升

结论：

不推荐。它只能把“英文机械摘要器”改造成“中文机械摘要器”。

## 2.2 方案 B：完全交给 LLM 端到端写长报告

思路：

- 后端只提供文件列表和若干源码片段
- 让模型直接读上下文并一次性生成 3000+ 字中文报告

优点：

- 写作自然度最高
- 实现表面上最快
- 容易得到更像“老师讲课”的风格

缺点：

- 容易幻觉
- 对大仓库和复杂仓库不稳
- 报告质量强依赖模型一次性发挥
- 很难稳定控制“必须覆盖哪些章节”和“不能机械引用”
- 后续回归测试会非常难做

结论：

也不推荐。它太依赖模型临场表现，不适合做成稳定产品能力。

## 2.3 方案 C：混合式流水线，先做证据与结构，再做中文教学写作

思路：

- 前半段用确定性分析器提取可信证据和结构
- 中间层构建“教学问题清单”和“章节 brief”
- 后半段只让模型负责中文教学写作与自然组织
- 最终通过质量门禁决定是否接受报告

优点：

- 既有证据基础，又有自然写作能力
- 可以强控制章节、长度、语言、引用风格
- 能把“为什么这样设计”与“模块如何协作”真正写出来
- 回归测试比纯 LLM 方案更可控
- 便于后续 agent 分模块搭脚手架

缺点：

- 设计和实现复杂度高于方案 A
- 需要新增若干内部数据结构与质量门禁模块

结论：

推荐采用。它是唯一能同时兼顾“稳定性、教学性、中文质量、证据约束、后续可维护性”的路线。

## 3. 推荐方案总览

推荐方案是：把 deepresearch 升级为“混合式中文教学长报告流水线”。

整体策略：

1. 继续保留当前 session、SSE、前端渲染方式。
2. deepresearch 内部不再直接用 `render_final_report()` 拼固定英文章节。
3. 先做确定性证据抽取，再做语义图谱和教学大纲。
4. 让模型按章节 brief 写中文内容，而不是让模型直接从零自由发挥。
5. 对最终报告做硬性质量门禁，不达标就重写或失败。

## 4. 解决方案的具体组成

## 4.1 第一层：证据层

这一层只负责“拿到真实可用的代码证据”，不负责写报告。

它要做的事：

- 读取仓库中的关键源码文件、配置文件和文档
- 同时覆盖后端与前端源码，不允许只读 Python 不读 `web/`
- 构建更细粒度的模块单元，而不是只按顶层目录分组
- 提取：
  - 入口候选
  - 模块边界
  - import 关系
  - 关键类与函数
  - 配置入口
  - 状态承载对象
  - API/路由/服务/存储等角色
- 给每条证据打上来源路径

这一层要避免的事：

- 不直接生成最终中文段落
- 不直接下结论说“架构就是 X”
- 不输出成型教学文案
- 不允许因为仓库主语言是 Python，就忽略 `web/` 中的 API 调用、状态管理和渲染链路

为什么这层必要：

- 因为教学报告必须有底座，否则长报告只会变成长 hallucination

## 4.2 第二层：语义与教学规划层

这一层是关键。它决定 deepresearch 最终像不像“老师讲课”。

它要把证据转成可回答的教学问题，例如：

- 这个项目最核心的运行链路是什么
- 系统主要由哪几块组成
- 模块边界是怎么划的
- 为什么 `SessionService`、`AnalysisWorkflow`、`ChatWorkflow` 要分开
- `contracts`、`m5_session`、`m6_response`、`agent_runtime` 之间的连接点是什么
- `web/js/api.js`、`state.js`、`views.js` 如何配合
- 当前实现中哪些结构是为了解耦，哪些是为了契约稳定

然后它要输出一份强约束大纲：

- 每章必须回答什么
- 每章必须引用哪些证据
- 每章不能写什么空话
- 每章的最小字数是多少
- 哪些结论只能写成“推断”

为什么这层必要：

- 因为“教学性”不是代码里直接长出来的，需要中间的讲解组织层

## 4.3 第三层：中文章节写作层

这一层才是 LLM 真正发挥的地方。

它不再接收“整个仓库”，而是接收：

- 当前章节目标
- 当前章节必须回答的问题
- 当前章节允许使用的证据
- 当前章节必须覆盖的模块与数据结构
- 当前章节的风格要求
- 当前章节的禁写项

这样可以显著降低：

- 英文漂移
- 空泛总结
- 机械引用
- 漏讲关键点

为什么这层能满足你的要求：

- 模型擅长自然写作和教学组织
- 但必须被限制在“基于证据、围绕问题、按章节写作”的轨道里

## 4.4 第四层：报告装配与质量门禁层

最终报告不是“章节拼起来就算完”，还要做质量检查。

必须检查：

- 中文比例是否足够高
- 正文字数是否达到 3000 汉字
- 是否覆盖全部强制章节
- 是否解释了“为什么这样设计”
- 是否解释了“怎么解耦，为什么解耦”
- 是否解释了“用了哪些数据结构，为什么”
- 是否出现大量路径清单堆砌
- 是否存在机械化的逐句证据罗列
- 是否把未验证内容说成了已验证事实

如果不通过：

- 自动进行一次定向重写
- 如果仍不通过，则返回明确失败原因，而不是默默给出一份不合格报告

## 5. 为什么这套方案能满足你的要求

### 5.1 能保证中文

因为中文不再依赖“模型碰巧输出中文”，而是：

- 章节 writer 明确要求简体中文
- 质量门禁会检查中文比例
- English heading 不再允许出现在最终可见正文模板中

### 5.2 能保证至少 3000 字

因为长度不再是“希望模型自己多写点”，而是：

- 每章设最小字数
- 总报告设最小汉字数
- 不达标直接判失败

### 5.3 能保证教学性

因为写作入口不是“总结仓库”，而是“回答教学问题”。

例如“为什么这样架构”这类问题，会被强制写进章节 brief，而不是靠模型临时想起。

### 5.4 能避免机械引用

因为引用风格会从“路径清单”改为“证据锚点”。

也就是：

- 正文先自然解释
- 小节末尾再列 2-5 个关键证据路径
- 禁止每段都堆一串文件路径

### 5.5 能讲清模块协作和解耦

因为中间层会显式构建：

- 模块图
- 依赖边
- 协作边
- 状态边
- 边界接口

而不是继续停留在“这个文件 import 了那个文件”。

### 5.6 能保持原版交互不变

因为外部接口保持不变：

- 仍然使用 `/api/repo`
- 仍然使用 `/api/analysis/stream`
- 仍然把正文渲染到 `raw_text`
- `web/` 只需要继续展示原始 Markdown

换句话说，真正要升级的是 deepresearch 的内部生成流水线，不是产品交互壳子。

## 6. 必须同步修改的设计原则

为了让后续 agent 不再被旧原则束缚，建议明确写入新原则：

1. `quick_guide` 继续遵守“backend 不直接产出成型架构结论”的保守规则。
2. `deep_research` 作为独立长报告模式，允许 backend 组织一套更强的证据分析与报告生成流水线。
3. `deep_research` 的所有成型结论仍然必须带证据来源与置信度，不允许无根据断言。

这三条一起成立，才能既保住原产品定位，又允许新功能真正满足你的需求。

## 7. 最小落地策略

建议采用三步推进，而不是一次性大改。

### 第一步：替换 deepresearch 内部流水线，不动外部交互

只改：

- `backend/deep_research/`
- `backend/m5_session/analysis_workflow.py`
- 对应测试

不改：

- `web/` 交互
- `/api/repo`
- `/api/analysis/stream`
- quick guide prompt 体系

### 第二步：补质量门禁和中文长报告回归测试

新增测试检查：

- 中文比例
- 强制章节
- 最小长度
- 关键问题覆盖率
- 非机械引用

### 第三步：再考虑是否需要 richer progress 或 sidecar explain

这一步不是必须的。正文质量优先，交互花活后置。

## 8. 推荐结论

推荐采用方案 C，也就是：

**保留原始交互与会话契约，只把 `deep_research` 升级为“证据抽取 + 语义建模 + 教学规划 + 中文章节写作 + 质量门禁”的混合式长报告流水线。**

这样做的结果是：

- quick guide 继续保守、轻量、可验证
- deepresearch 变成真正可用的教学长报告能力
- 后续 agent 也有清晰模块边界可接手
