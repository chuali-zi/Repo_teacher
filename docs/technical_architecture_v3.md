## <a id="arch-01"></a>ARCH-01 系统总览与技术选型

### 架构范式

**前后端分离的单体 Web 应用**，后端为 Python，前端为 SPA。

- 用户通过浏览器访问前端页面
- 前端通过 HTTP/SSE 与后端通信
- 后端负责仓库接入、静态分析、教学骨架组装、对话管理、LLM 调用
- 第一版为单用户本地部署（`localhost`），不考虑多租户

### 技术栈

| 层级 | 技术选型 | 理由 |
|------|---------|------|
| Python 运行时 | Python 3.11+ | 标准库 `tomllib` 可解析 `pyproject.toml`；`sys.stdlib_module_names` 可支持标准库 import 判定 |
| Node 运行时 | Node.js 20 LTS+ | React + TypeScript 本地开发基线，避免脚手架生成过旧工具链 |
| 后端框架 | FastAPI | 原生 async 支持，适合流式输出；Python 生态与分析模块一致 |
| 前端框架 | React + TypeScript | 组件化适合对话式 UI，社区生态成熟 |
| 前后端通信 | HTTP REST + SSE（Server-Sent Events） | REST 用于仓库提交/状态查询；SSE 用于分析进度推送和回答流式输出 |
| Python AST 分析 | Python 标准库 `ast` | 零依赖，满足第一版 Python 静态分析需求 |
| LLM 调用 | DeepSeek API（OpenAI 兼容接口，`openai` SDK） | 兼顾成本、响应速度与接入通用性，降低对单一供应商 SDK 的耦合 |
| GitHub 克隆 | `git` CLI 调用（subprocess） | 简单可靠，shallow clone 控制体积 |
| 进程管理 | `uvicorn` | FastAPI 标准 ASGI server |
| 包管理 | `uv` 或 `pip` | 快速依赖解析 |

### 部署形态

```
用户本机
┌─────────────────────────────────────────────┐
│  Browser (localhost:3000)                   │
│     ↕ HTTP/SSE                              │
│  FastAPI Backend (localhost:8000)            │
│     ↕ subprocess                            │
│  git CLI（克隆 GitHub 仓库到临时目录）         │
│     ↕ DeepSeek API                          │
│  DeepSeek API (外部)                        │
└─────────────────────────────────────────────┘
```

第一版为本地运行，`npm run dev` + `uvicorn` 同时启动前后端。

## <a id="arch-03"></a>ARCH-03 模块 M1 — 仓库接入层

### 职责

1. 接收用户输入的仓库路径或 GitHub URL
2. 格式校验（路径格式 / URL 格式）
3. 可访问性验证（路径存在且为目录 / GitHub 仓库公开可访问）
4. GitHub 仓库克隆到本地临时目录（shallow clone）
5. 输出统一的仓库根目录路径供下游使用

### 对应 PRD/交互设计

- PRD: INPUT 章节、F1 仓库只读接入
- 交互: IX-03 首次仓库接入、IX-07 输入交互规格、IX-09 异常与降级交互

### 内部子模块

| 子模块 | 职责 |
|--------|------|
| `input_validator` | 输入格式校验：区分本地路径 vs GitHub URL，校验格式合法性 |
| `local_repo_accessor` | 本地仓库验证：路径存在、是目录、可读 |
| `github_repo_cloner` | GitHub 仓库克隆：URL 解析、`git clone --depth=1` 到临时目录、失败处理 |

### 输入

- 用户输入字符串（本地路径或 GitHub URL）

### 输出

- 统一的仓库访问描述（仓库根路径、来源类型、仓库名称、是否为临时目录）
- 或错误描述（错误类型 + 面向用户的提示文案）

### 关键行为

- GitHub 克隆使用 `--depth=1` 限制体积
- 临时目录在会话结束或切换仓库时清理
- 路径越界防护：本地路径必须为绝对路径，禁止 `..` 遍历（详见 ARCH-13）

### 依赖

- 外部：`git` CLI
- 内部：无（最底层模块之一）

## <a id="arch-04"></a>ARCH-04 模块 M2 — 文件树扫描与过滤

### 职责

1. 递归扫描仓库目录，构建文件树
2. 应用敏感文件黑名单过滤（PRD INPUT 章节）
3. 应用忽略规则（`.gitignore`、`__pycache__`、`.git`、`node_modules` 等）
4. 计算仓库规模指标（源码文件数），判定是否触发降级
5. 识别主语言（基于文件扩展名统计）

### 对应 PRD/交互设计

- PRD: INPUT 安全边界、F2 文件树扫描、NFR 性能目标
- 交互: IX-09 降级交互

### 内部子模块

| 子模块 | 职责 |
|--------|------|
| `tree_scanner` | 递归遍历目录，构建文件树结构 |
| `file_filter` | 应用敏感文件黑名单 + 忽略规则，标记文件状态（正常/敏感跳过/忽略） |
| `repo_sizer` | 统计源码文件数，判定规模级别（小型 <500 / 中型 500-3000 / 大型 >3000） |
| `language_detector` | 基于扩展名统计主语言，判定是否为 Python 仓库 |

### 输入

- M1 输出的仓库根路径

### 输出

- 文件树（含每个节点的路径、类型、状态标记）
- 仓库规模级别
- 主语言判定
- 敏感文件清单（仅记录存在，不含内容）

### 关键行为

- 敏感文件黑名单匹配模式：`.env`, `.env.*`, `*.pem`, `*.key`, `*.crt`, `id_rsa`, `id_ed25519`, `credentials*`, `secrets*`, `token*`
- 命中敏感文件时：记录"文件存在但未读取内容"，后续模块不得读取其正文
- 大型仓库（>3000 源码文件）时：标记降级模式，M3 分析引擎将限制扫描范围

### 依赖

- 内部：M1（仓库根路径）

## <a id="arch-05"></a>ARCH-05 模块 M3 — 静态分析引擎

### 职责

这是系统的核心分析模块。职责是从仓库文件中提取教学所需的结构化分析产物。

**PRD ANALYSIS 章节明确要求**：分析模块不能只产出"目录摘要 + 入口启发式 + import 分类"，必须支持教学骨架的组装。

### 对应 PRD/交互设计

- PRD: ANALYSIS 章节、F2-F6/F8-F11、OUT-1 至 OUT-8、EVIDENCE 章节
- 交互: IX-03（分析步骤展示）

### 内部子模块

| 子模块 | 职责 | PRD 对应 |
|--------|------|---------|
| `entry_detector` | 入口候选识别 | OUT-4, F5 |
| `import_analyzer` | import 来源分类（内部/标准库/第三方/未知） | OUT-5, F6 |
| `module_identifier` | 关键目录/模块识别与职责推断 | OUT-2, OUT-8, F11 |
| `layer_inferrer` | 教学式分层推断 | OUT-7, F10 |
| `flow_tracer` | 候选数据流/主流程骨架追踪 | OUT-6, F9 |
| `project_profiler` | 项目类型候选判定 | F4 |
| `reading_path_builder` | 阅读路径生成 | OUT-3, F8 |
| `evidence_collector` | 证据收集与置信度评估 | EVIDENCE 章节 |

### 子模块详细说明

#### `entry_detector` — 入口候选识别

**检查来源**（按 PRD OUT-4）：
- `__main__.py`
- `main.py`, `app.py`, `manage.py`
- 文件内 `if __name__ == "__main__"` 块
- `pyproject.toml` 中的 `[project.scripts]` / `[tool.poetry.scripts]`
- `setup.py` 中的 `entry_points`
- README 中的运行命令
- 框架约定入口（Flask `app = Flask(...)`, FastAPI `app = FastAPI(...)`, Django `manage.py`）

**输出**：入口候选列表，每个含：候选文件/命令、理由、置信度、排序依据

**约束**：找不到时输出"未知"，不得硬猜

#### `import_analyzer` — import 来源分类

**工作方式**：
1. 用 `ast` 模块解析 Python 文件，提取所有 `import` 和 `from ... import` 语句
2. 逐条判断来源：
   - 项目内部模块：在仓库文件树中找到对应路径
   - Python 标准库：与 `sys.stdlib_module_names`（Python 3.10+）或内置标准库列表比对
   - 第三方库：在 `requirements.txt` / `pyproject.toml` / `setup.py` / `Pipfile` 的依赖声明中找到
   - 未知：以上均不命中

**输出**：import 来源分类列表，每条含：import 名称、来源判断、判断依据、使用该 import 的文件列表

**约束**：不得把未知强行归类为第三方

#### `module_identifier` — 关键目录/模块识别

**识别逻辑**：
1. 从文件树中识别顶层目录和含 `__init__.py` 的 Python 包
2. 结合 import 被引用频次、文件数量、入口是否直接引用来判定重要性
3. 推断模块职责（基于目录名、文件名、内部 import 关系、README 提及）

**输出**：关键模块列表，每个含：路径、推断职责、重要性排序、依据

#### `layer_inferrer` — 教学式分层推断

**工作方式**：
1. 基于已识别的入口候选、模块、import 关系，将模块映射到教学层次
2. 允许的层次（PRD OUT-7）：入口层、路由/控制层、业务逻辑层、数据访问层、工具/配置层
3. 映射依据：入口文件 → 入口层；路由/视图文件 → 路由层；含 DB/ORM 操作 → 数据层；utils/helpers → 工具层
4. 证据不足时标注"启发式分层"或"未知"

**输出**：分层视图，每层含：层名称、对应模块/目录候选、角色说明、是否主路径、依据

#### `flow_tracer` — 候选数据流/主流程骨架追踪

**工作方式**（PRD OUT-6 深度分级）：
1. 从入口候选出发，追踪入口文件的顶层 import 和直接函数调用
2. 基于 import 关系、配置文件、README 线索，推断模块级调用路径
3. 结合分层结果，组织为"输入从哪进 → 经过哪些层 → 结果去向"的教学骨架
4. 对每个步骤标注置信度

**输出**：候选流程列表，每条含：入口文件、输入来源、模块级路径、层级经过、结果去向、置信度、不确定项

**约束**：不要求证明真实运行时调用链；证据不足时降级为阅读建议

#### `project_profiler` — 项目类型候选判定

**判定依据**：
- 框架依赖（Flask/FastAPI/Django → Web 项目；Click/argparse → CLI 项目）
- 项目结构特征（`setup.py`/`pyproject.toml` 的 package 配置 → library）
- README 中的项目描述

**输出**：项目类型候选 + 置信度

#### `reading_path_builder` — 阅读路径生成

**工作方式**：
1. 综合入口候选、分层、流程骨架、模块重要性
2. 生成 3-6 步阅读路径
3. 每步含：目标、原因、学习收益、可跳过内容、下一步、证据

**依赖**：依赖 `entry_detector`, `module_identifier`, `layer_inferrer`, `flow_tracer` 的输出

#### `evidence_collector` — 证据收集与置信度评估

**职责**：
- 为所有分析子模块提供统一的证据记录机制
- 每条证据含：类型（文件路径/README/配置/import/符号/目录结构/命名约定）、内容、来源位置
- 置信度评估：高/中/低/未知（按 PRD EVIDENCE 章节的分级标准）

### M3 整体运行顺序

```
文件树 (M2输出)
    │
    ▼
project_profiler ──→ 项目类型候选
    │
    ▼
entry_detector ──→ 入口候选集合
    │
    ├─→ import_analyzer ──→ import 来源分类
    │
    ├─→ module_identifier ──→ 关键模块集合
    │         │
    │         ▼
    ├─→ layer_inferrer ──→ 教学式分层视图
    │         │
    │         ▼
    └─→ flow_tracer ──→ 候选流程骨架
              │
              ▼
    reading_path_builder ──→ 阅读路径
```

注意：`import_analyzer` 和 `module_identifier` 可并行执行；`layer_inferrer` 依赖 `module_identifier`；`flow_tracer` 依赖 `entry_detector` + `module_identifier` + `layer_inferrer`；`reading_path_builder` 依赖以上所有。

### 非 Python 仓库降级

当 M2 判定主语言非 Python 时，M3 仅运行：
- `module_identifier`（降级为基于目录名的保守推断）
- 不运行 `entry_detector`（Python 特定）、`import_analyzer`（Python 特定）、`layer_inferrer`、`flow_tracer`
- `reading_path_builder` 仅基于目录结构给出保守建议

### 大型仓库降级

当 M2 判定为大型仓库（>3000 文件）时：
- `import_analyzer` 仅分析入口文件及其直接 import 的文件
- `flow_tracer` 仅追踪入口附近 1-2 层
- `reading_path_builder` 优先输出结构总览和阅读起点

### 依赖

- 内部：M2（文件树、规模、主语言）
- 外部：Python `ast` 标准库

## <a id="arch-06"></a>ARCH-06 模块 M4 — 教学骨架组装器

### 职责

将 M3 的零散分析产物组装为**教学可用骨架**，使其可被 M6 回答生成器按主题灵活引用。

**PRD ANALYSIS 章节明确要求**：分析结果不能硬编码进单一 prompt 文案，每个结论应可被回答层按主题重组。

### 对应 PRD/交互设计

- PRD: ANALYSIS 组织原则、OUT-1 首轮教学报告的内容组织
- 交互: IX-08 输出内容展示规格

### 内部子模块

| 子模块 | 职责 |
|--------|------|
| `skeleton_assembler` | 将分析产物按"观察框架 → 仓库映射 → 阅读起点"顺序组织为首轮教学骨架 |
| `topic_indexer` | 为每个分析产物建立主题索引（结构/入口/流程/分层/依赖/模块），使回答层可按主题抽取 |
| `unknown_aggregator` | 汇总所有"未知"项和"不确定"项，形成统一的未知项集合 |

### 输入

- M3 全部分析产物（入口候选、import 分类、模块集合、分层视图、流程骨架、阅读路径、项目画像、证据集合）

### 输出

- 首轮教学骨架（按 OUT-1 结构组织）
- 主题索引（结构/入口/流程/分层/依赖/模块 → 对应分析产物的引用）
- 未知项集合

### 关键行为

- 首轮骨架严格按 PRD OUT-1 的字段顺序组织：仓库概览 → 先抓什么 → 仓库映射 → 语言与类型 → 关键目录 → 入口候选 → 推荐第一步 → 阅读路径 → 不确定项
- 主题索引支持 M6 在多轮对话中按用户追问主题抽取相关内容
- 非 Python 降级时组装降级骨架（仅文件树总览 + 保守阅读建议 + 降级标注）

### 依赖

- 内部：M3（分析产物）

## <a id="arch-07"></a>ARCH-07 模块 M5 — 对话管理器

### 职责

管理多轮教学对话的全部状态，是后端的协调中心。

### 对应 PRD/交互设计

- PRD: OUT-9 多轮回答状态、OUT-10 学习目标切换、F12 多轮上下文保持、F13 学习目标切换、F16 回答深浅控制
- 交互: IX-04 多轮教学对话、IX-05 学习目标切换、IX-06 状态定义与转换、IX-10 深浅控制

### 内部子模块

| 子模块 | 职责 |
|--------|------|
| `session_store` | 维护当前会话状态（当前仓库、分析结果缓存、对话历史） |
| `intent_classifier` | 识别用户消息意图：追问/目标切换/深浅调整/切换仓库/阶段性总结 |
| `state_machine` | 管理全局状态转换（空闲态→接入中→分析中→对话态）和对话子状态 |
| `depth_controller` | 管理深浅级别（浅/默认/深），记录切换历史 |

### 会话状态（PRD OUT-9 明确要求保持的状态）

```
当前仓库
当前学习目标
当前讲解阶段
当前聚焦模块
当前入口候选
当前候选流程
当前分层理解结果
已讲解过的文件和模块
上一轮下一步建议
当前深浅级别
对话历史（用于 LLM 上下文）
```

### 意图分类

| 意图类型 | 触发条件 | 后续动作 |
|----------|---------|---------|
| 自由追问 | 默认，用户提出关于当前仓库的问题 | 将问题 + 会话状态 + 教学骨架 → M6 生成回答 |
| 目标切换 | 匹配"只看XX"模式 | 更新学习目标，将目标主题 + 教学骨架对应切片 → M6 |
| 深浅调整 | 匹配"讲浅/深一点" | 更新深浅级别，后续 M6 调整 prompt 参数 |
| 阶段性总结 | 匹配"总结一下" | 将对话历史 + 已讲解内容 → M6 生成总结 |
| 切换仓库 | 用户点击"切换仓库" | 清理当前会话，回到空闲态 |

### 意图分类实现方式

第一版使用**规则匹配优先 + LLM 兜底**：
1. 先用关键词/正则匹配已知意图模式（"只看入口"、"讲浅一点"、"总结一下"等）
2. 未匹配到已知模式时，默认为自由追问，由 M6 在 LLM prompt 中处理

### 全局状态机

```
空闲态 ──[用户提交仓库]──→ 接入中
接入中 ──[接入成功]──→ 分析中
接入中 ──[接入失败]──→ 接入错误 ──[用户修改重试]──→ 接入中
分析中 ──[分析成功]──→ 对话态
分析中 ──[分析失败]──→ 分析错误 ──[用户修改重试]──→ 空闲态
对话态 ──[用户切换仓库]──→ 空闲态
```

### 关键行为

- 第一版为单会话，内存中维护状态，不持久化
- 切换仓库时清理全部会话状态（含临时克隆目录）
- 深浅级别在切换仓库后重置为"默认"

### 依赖

- 内部：M1（仓库接入）、M2（文件树扫描）、M3（静态分析）、M4（教学骨架组装）、M6（回答生成）

## <a id="arch-08"></a>ARCH-08 模块 M6 — 回答生成器

### 职责

基于教学骨架和会话状态，调用 LLM 生成符合 PRD 教学规则的自然语言回答。

### 对应 PRD/交互设计

- PRD: OUT-1 至 OUT-12、UX 规则 1-5、EVIDENCE 候选措辞
- 交互: IX-04 回答结构、IX-08 输出展示规格、IX-10 深浅控制

### 内部子模块

| 子模块 | 职责 |
|--------|------|
| `prompt_builder` | 根据场景（首轮报告/追问回答/目标切换/阶段性总结）构建 LLM prompt |
| `llm_caller` | 封装 DeepSeek API 调用，支持流式输出 |
| `response_parser` | 解析 LLM 返回，提取结构化字段（本轮重点/直接解释/与整体关系/证据/不确定项/下一步建议） |
| `suggestion_generator` | 基于当前讲解进度和已讲内容，生成 1-3 条下一步建议 |

### Prompt 构建策略

#### 首轮报告 prompt

```
系统角色设定（教学 Agent 身份、教学规则）
+ 教学骨架全文（M4 输出的首轮教学骨架）
+ 输出格式约束（按 OUT-1 字段顺序、候选措辞、置信度标注）
+ 深浅级别（默认）
```

#### 多轮追问 prompt

```
系统角色设定
+ 教学骨架的主题切片（根据意图从 M4 主题索引中抽取相关部分）
+ 会话上下文（近 N 轮对话历史摘要 + 当前学习目标 + 已讲解内容列表）
+ 用户当前问题
+ 输出格式约束（OUT-11 六段式结构）
+ 深浅级别
```

#### 阶段性总结 prompt

```
系统角色设定
+ 已讲解内容列表
+ 近 N 轮对话历史
+ 输出格式约束（总结结构：已了解/未展开/建议下一步）
```

### 流式输出

- LLM 调用使用流式 API（`stream=True`）
- 后端通过 SSE 逐 chunk 推送给前端
- 前端实时渲染，无需等待完整回答

### DeepSeek 接入约束

- `llm_caller` 通过 OpenAI 兼容客户端接入 DeepSeek，使用 `base_url=https://api.deepseek.com`
- 第一版默认使用 `deepseek-chat` 作为教学回答模型，优先保证响应速度和成本可控
- 模型名、`base_url`、API Key 由配置层注入，避免在业务逻辑中写死供应商参数

### 下一步建议生成

- 基于当前已讲解内容和教学骨架中未讲解的主题
- 排除已讲解过的内容
- 优先推荐教学主线中的下一个自然步骤

### 依赖

- 内部：M4（教学骨架）、M5（会话状态）
- 外部：DeepSeek API（OpenAI 兼容接口）

## <a id="arch-11"></a>ARCH-11 运行时流程

### 流程一：首次仓库接入（对应 IX-03）

接口边界以 `interface_hard_spec_v3.md` 为准：`POST /api/repo` 只同步完成请求参数校验、旧会话清理、新会话创建和异步流程启动，成功返回 `202 Accepted`。请求体缺字段、空字符串或格式无法解析属于同步 `400 Bad Request`；M1 可访问性验证、clone、扫描、分析和首轮报告生成属于异步流程，失败必须通过 `/api/analysis/stream` 的 `error` 事件返回。

```
用户提交仓库路径/URL
        │
        ▼
[路由层 / M5] 同步请求参数校验
        ├── 参数不合法 → HTTP 400（直接返回）
        │
        └── 参数合法
                │
                ▼
            [M5] 状态 → 接入中，并初始化进度快照
                │
                ▼
            [M1] 可访问性验证（本地:路径存在/可读；GitHub:公开可访问 + clone）
                │
                ├── 失败 → SSE错误
                │
                └── 成功
                    │
                    ▼
                [M5] 状态 → 分析中
                    │
                    ▼
                [M2] 文件树扫描 + 过滤 + 规模判定 + 语言检测
                ──→ SSE 推送: "文件树扫描完成"
                    │
                    ▼
                [M3] 静态分析引擎（按子模块依赖顺序执行）
                ──→ SSE 推送: "正在识别入口与模块..."
                ──→ SSE 推送: "正在分析依赖来源..."
                    │
                    ▼
                [M4] 教学骨架组装
                ──→ SSE 推送: "正在组织教学骨架..."
                    │
                    ▼
                [M6] 首轮报告生成（LLM 调用，流式输出）
                ──→ SSE 推送: 首轮报告内容（流式）
                    │
                    ▼
                [M5] 状态 → 对话态
                [M5] 缓存分析结果、教学骨架和最终进度快照到会话状态
```

说明：`[M5] 状态 → 接入中，并初始化进度快照` 只发生在同步参数校验成功之后；参数校验失败直接返回 `HTTP 400`，不会创建或推进异步接入流程。

## <a id="arch-12"></a>ARCH-12 降级与异常策略

### 降级场景

| 场景 | 触发条件 | 降级行为 | 对应 PRD/交互 |
|------|---------|---------|-------------|
| 大型仓库 | M2 判定源码文件 >3000 | M3 限制分析范围；首轮报告标注"部分内容因规模限制未完成" | NFR 性能、IX-09 |
| 非 Python 仓库 | M2 判定主语言非 Python | M3 仅运行基于目录的保守分析；M4 组装降级骨架；M6 使用降级报告模板 | OUT-12、IX-09 |
| 入口未找到 | M3 entry_detector 无可靠候选 | 输出"未找到可靠入口候选"，给出替代阅读建议 | OUT-4 |
| 流程无法形成 | M3 flow_tracer 证据不足 | 输出"证据不足以形成可靠流程"，给出阅读建议替代 | OUT-6 |
| 分层无法稳定 | M3 layer_inferrer 证据不足 | 标注"启发式分层"或"当前未知" | OUT-7 |

### 异常处理

| 异常场景 | 处理策略 | 用户感知 |
|---------|---------|---------|
| 本地路径不存在 | M1 返回错误 | 视图 A 红色提示 |
| 本地路径不是目录 | M1 返回错误 | 视图 A 红色提示 |
| GitHub URL 格式非法 | M1 返回错误 | 视图 A 红色提示 |
| GitHub 仓库不可访问（404/403） | M1 clone 失败，返回错误 | 视图 A 红色提示 |
| Git clone 超时 | M1 设置超时（30s），超时返回错误 | 视图 A 红色提示 |
| AST 解析单文件失败 | M3 跳过该文件，记录警告，继续分析 | 不中断，首轮报告不确定项中可能提及 |
| LLM API 调用失败 | M6 重试 1 次；仍失败则 M5 向用户报错 | 对话视图显示错误提示 |
| LLM API 超时 | M6 设置超时（60s），超时后报错 | 对话视图显示超时提示 |
| 分析整体超时 | M5 设置总超时（小型30s/中型90s），超时启用降级 | 分析视图显示降级提示 |

### 错误返回原则

- 所有错误返回面向用户的自然语言提示，不暴露技术堆栈
- 错误不清空用户已输入内容
- 分析过程中的非致命错误（如单文件 AST 解析失败）不中断整体流程

## <a id="arch-13"></a>ARCH-13 安全架构

### 只读策略

| 约束 | 实现方式 |
|------|---------|
| 不执行仓库代码 | M3 仅使用 `ast.parse()` 解析，不使用 `exec`/`eval`/`importlib` |
| 不修改仓库文件 | 所有文件操作仅 `open(path, 'r')`，不使用写模式 |
| 不安装依赖 | 不调用 `pip install`，依赖判断仅基于声明文件内容 |
| 不执行 shell 命令 | 除 `git clone` 外，不调用任何 subprocess |

### 敏感文件保护

- M2 文件树扫描时，对命中黑名单的文件标记为 `sensitive_skipped`
- M3 分析引擎在读取文件内容前，检查文件状态标记，遇到 `sensitive_skipped` 跳过
- M6 回答生成器的 prompt 中不包含任何敏感文件内容
- 输出中仅说明"该文件因安全策略未读取内容"

### 路径越界防护

- M1 接收本地路径后，解析为绝对路径（`os.path.realpath`）
- M2 扫描时，所有文件路径必须以仓库根路径为前缀
- 禁止符号链接指向仓库外部（`os.path.realpath` 后检查前缀）

### GitHub 安全

- 第一版仅支持公开仓库（`https://github.com/owner/repo`）
- 不携带任何认证 token 克隆
- Clone 到系统临时目录，会话结束后清理

## <a id="arch-14"></a>ARCH-14 非功能约束与性能设计

### 性能目标（PRD NFR）

| 仓库规模 | 源码文件数 | 目标 | 实现策略 |
|----------|-----------|------|---------|
| 小型 | <500 | 30 秒内首轮输出 | 全量分析，无需降级 |
| 中型 | 500-3000 | 90 秒内首轮输出 | 全量分析，SSE 阶段性反馈 |
| 大型 | >3000 | 明确告知降级 | 限制分析范围，优先输出结构总览 |

### 分析阶段性反馈

M5 在分析过程中通过 SSE 向前端推送步骤进度：
1. 仓库接入成功 ✓
2. 文件树扫描完成 ✓
3. 正在识别入口与模块 ●
4. 分析依赖来源 ○
5. 组织教学骨架 ○

前端根据推送更新进度步骤清单状态（对应 IX-02 视图 B）。

### 分析结果缓存

- 同一会话内，M3 分析结果和 M4 教学骨架缓存在 M5 的会话状态中
- 多轮追问不重新触发分析（除非切换仓库）
- 第一版不做磁盘持久化缓存

### LLM 调用优化

- 首轮报告的 prompt 较长（含完整教学骨架），通过复用固定 system prompt 和会话内教学骨架缓存，减少重复拼装与重复输入
- 多轮追问时，仅注入主题切片、近 N 轮摘要和当前学习目标，不依赖供应商专有 prompt caching
- 流式输出减少用户感知延迟

### 可测试性

- 每个 M3 子模块可独立单元测试（输入文件树/文件内容，输出分析结果）
- M4 可独立测试（输入 mock 分析产物，输出教学骨架）
- M6 可用 mock LLM 测试 prompt 构建逻辑
- 端到端测试使用 PRD ACCEPTANCE 中定义的 3 个固定样例仓库

## <a id="arch-15"></a>ARCH-15 架构决策记录

### ADR-01：前后端分离 vs 纯 CLI

**决策**：前后端分离 Web 应用

**理由**：
- PRD 和交互设计定义了视图 A/B/C、消息流、可点击建议按钮等丰富交互
- 纯 CLI 无法满足 IX-02 的视觉布局要求
- Web 前端天然支持 Markdown 渲染、流式输出、按钮交互

### ADR-02：SSE vs WebSocket

**决策**：使用 SSE（Server-Sent Events）

**理由**：
- 数据流为单向（服务端 → 客户端推送进度和回答）
- 客户端 → 服务端通信用普通 HTTP POST 即可
- SSE 实现更简单，不需要双向通道
- FastAPI 原生支持 `StreamingResponse`

### ADR-03：M3 内部子模块 vs LLM 直接分析

**决策**：M3 使用基于规则和 AST 的确定性分析，不依赖 LLM

**理由**：
- PRD 要求确定性结论必须有证据，LLM 分析会引入幻觉风险
- 入口识别、import 分类、文件树分析等任务有明确的规则可循
- LLM 仅在 M6 回答生成阶段使用，负责将结构化数据转化为教学自然语言
- 分析产物的可测试性和可复现性更高

### ADR-04：意图分类策略

**决策**：规则匹配优先 + LLM 兜底

**理由**：
- PRD 定义的目标切换指令模式有限且明确（"只看XX"、"讲浅/深一点"、"总结一下"）
- 规则匹配速度快、可预测
- 未匹配的自由追问交给 LLM 在 prompt 中自然处理，无需额外分类模型

### ADR-05：单会话内存 vs 持久化存储

**决策**：第一版使用内存存储，不引入数据库

**理由**：
- 第一版为单用户本地部署
- 会话状态在切换仓库或关闭应用时清除即可
- 避免引入不必要的基础设施复杂度
- P2 的后续实验预留可在未来引入持久化

### ADR-06：教学骨架组装器独立成模块（M4）

**决策**：M4 独立于 M3 和 M6

**理由**：
- PRD ANALYSIS 明确要求"分析结果不能硬编码进单一 prompt 文案，每个结论应可被回答层按主题重组"
- M3 产出原始分析产物，M4 将其组织为教学结构，M6 将教学结构转化为自然语言
- 三者解耦使得：M3 可独立测试分析正确性；M4 可独立测试组装逻辑；M6 可用 mock 骨架测试 prompt

### ADR-07：Claude API 切换为 DeepSeek API

**决策**：M6 的外部 LLM 供应商从 Anthropic Claude API 切换为 DeepSeek API，并通过 OpenAI 兼容接口接入。

**理由**：
- 当前产品仅要求高质量教学回答、流式输出和稳定 API，不依赖 Claude 专有能力
- OpenAI 兼容接口可减少 SDK 适配复杂度，便于后续切换模型或保留多供应商扩展空间
- 供应商切换后，M1-M5 的确定性分析模块和 M7 的前端流式交互无需改动，影响面集中在 M6
