## <a id="wl-01"></a>WL-01 问题定义

当前 Repo Tutor 已经能：

- 基于 M4 教学骨架回答
- 维持基本消息历史
- 生成首轮报告和多轮回答

但它还不够像“老师”，主要缺 4 件事：

1. 没有显式维护“我这位老师现在在教什么、为什么这么教、下一步准备怎么带”的工作台状态。
2. 没有显式维护“这堂课的教学计划”并随对话更新。
3. 没有显式维护“学生现在学到哪、哪里可能没听懂、应该补什么”。
4. 没有独立的回合后更新器，把本轮输入和输出沉淀成下一轮可复用的教学状态。

目标不是把系统做成复杂多 Agent 平台，而是在现有单 Agent 结构里补上最小可用的“持续教学状态”。

---

## <a id="wl-02"></a>WL-02 当前状态

### 当前已具备

- 静态分析事实来源：M1-M4
- 会话消息历史：M5 `conversation.messages`
- 已讲解项：M5 `explained_items`
- 学习目标与深浅：M5 `current_learning_goal`、`depth_level`
- prompt 中的轻量上下文：`history_summary`、`topic_slice`、`teacher_memory`、`teaching_plan`

### 当前缺失

- `teacher_working_log`
- `teaching_plan_state`
- `student_learning_state`
- `state_updater`
- 以上状态的流式可视化或调试日志

### 当前判断

系统已经从“纯静态骨架复述”迈向“有老师口吻的回答”，但还没有进入“像老师连续备课和带课”的阶段。

---

## <a id="wl-03"></a>WL-03 目标形态

理想中的回合不只是：

1. 用户提问
2. LLM 回答

而应该更接近：

1. 老师读取当前教学状态
2. 老师判断这轮该推进哪一步教学计划
3. 老师估计学生现在的理解状态
4. 老师生成本轮回答
5. 系统把本轮结果写回教学状态
6. 下一轮继续沿着这堂课推进，而不是重新开始

这里的关键不是“工具调用”，而是“持续维护内部教学状态”。

## <a id="wl-04"></a>WL-04 教师 Working Log

### 定义

教师 Working Log 是老师在每一轮内部维护的一份工作台记录。它不是最终给学生看的正文，而是老师的“备课板”和“当前思路”。

### 目标

- 让老师知道自己正在教什么
- 让老师知道为什么当前要这么组织内容
- 让老师在回答前先看一眼“当前课堂局面”
- 让后续 Agent 或开发者能调试教学行为，而不是只看到最终回答

### 建议内容

建议至少包含以下字段：

- `current_teaching_objective`
- `why_now`
- `active_topic_refs`
- `current_plan_step_id`
- `planned_transition`
- `student_risk_notes`
- `recent_decisions`
- `open_questions`

### 设计原则

- 它应该是轻量的，不应成为第二套复杂知识库
- 它应该面向“下一轮如何教”，不是重复保存全量历史
- 它应该允许不确定，不假装自己知道学生真实心理状态

## <a id="wl-05"></a>WL-05 教学计划表

### 定义

教学计划表是一张面向当前仓库的、可更新的教学路线表。它不是死板脚本，而是老师当前认可的带读路线。

### 作用

- 首轮报告时给出“这节课先怎么走”
- 多轮追问时作为默认推进路线
- 当用户换目标时允许局部改写，而不是整张表失效

### 推荐结构

每一项计划至少应包含：

- `step_id`
- `title`
- `goal`
- `target_scope`
- `reason`
- `expected_learning_gain`
- `status`
- `priority`
- `depends_on`
- `source_topic_refs`
- `adaptation_note`

### 推荐状态

- `planned`
- `active`
- `completed`
- `deferred`
- `cancelled`

### 计划生成原则

- 初始计划应主要来自 M4 的 `reading_path_preview`、`focus_points`、`recommended_first_step`
- 计划不是简单复制阅读路径，而是把阅读路径转成“教学动作”
- 计划项要体现教学节奏：整体框架、入口、主流程、模块、细节

### 计划更新原则

- 用户显式切换目标时，可局部改写计划顺序
- 如果学生连续追问某一主题，可把对应计划项提到前面
- 如果某项证据不足，应标记 `deferred` 而不是强行完成

## <a id="wl-06"></a>WL-06 学生学习状态表

### 定义

学生学习状态表不是心理分析模型，而是一张保守的、教学用途的“学习进度估计表”。

### 为什么需要

老师感很大一部分来自：

- 知道学生已经听过什么
- 知道哪些地方可能还没建立稳固认知
- 知道下一轮应该补框架、补证据，还是继续深入

### 推荐结构

每个主题维度建议至少维护：

- `topic`
- `coverage_level`
- `confidence_of_estimate`
- `last_explained_at_message_id`
- `student_signal`
- `likely_gap`
- `recommended_intervention`
- `supporting_evidence`

### 推荐状态粒度

不要做得过细。建议先用这几档：

- `unseen`
- `introduced`
- `partially_grasped`
- `needs_reinforcement`
- `temporarily_stable`

### 重要提醒

这张表只能表达“教学上的保守估计”，不能表达“系统知道学生真的理解了”。

如果做得太像心智读取，会引入伪确定性。这里要坚持启发式、保守、可回退。

## <a id="wl-07"></a>WL-07 State Updater

### 定义

State Updater 是每轮结束后运行的一次轻量更新动作。它不负责回答用户，而是负责把本轮结果写回教学状态。

### 它应该更新什么

1. 更新教师 Working Log
2. 更新教学计划表
3. 更新学生学习状态表
4. 更新已讲解项和下轮推荐方向

### 它的输入

- 当前用户问题
- 当前回答的结构化结果
- 当前回答的原始正文摘要
- 当前计划表
- 当前学生状态表
- 当前已讲解项

### 它的输出

- 新版 `teacher_working_log`
- 新版 `teaching_plan_state`
- 新版 `student_learning_state`
- 变更说明 `update_notes`

### 设计原则

- updater 应是保守写入，不因一次模糊提问就大改整个教学路线
- updater 应优先基于结构化结果和明确用户信号
- updater 应允许“无更新”

## <a id="wl-08"></a>WL-08 回合循环

建议的最小回合循环如下：

1. M4 产出教学骨架
2. Plan Generator 基于骨架生成首版教学计划表
3. Student State Bootstrapper 生成首版学生状态表
4. Teacher Working Log 初始化当前教学目标与第一步动作
5. M6 在回答时读取：骨架 + 工作日志 + 教学计划 + 学生状态 + 会话历史
6. 回答完成后，State Updater 更新三张表
7. 下一轮再基于更新后的状态继续教学

核心思想是：

- 回答前先看“当前课堂状态”
- 回答后把“本轮教学结果”写回课堂状态

## <a id="wl-09"></a>WL-09 流式工作日志设计

### 目标

你提到的“流式教师 working log”是有价值的，但建议先定义为内部调试/可观测能力，而不是立刻暴露给最终用户。

### 推荐事件粒度

可考虑流式记录这些内部事件：

- `teacher_turn_started`
- `teaching_plan_selected`
- `teaching_plan_updated`
- `student_state_updated`
- `working_log_updated`
- `next_transition_selected`

### 每条日志建议回答的问题

- 老师这轮准备教什么
- 这个决定来自哪条计划或哪条用户信号
- 哪个学生状态判断影响了本轮讲法
- 下一轮最可能往哪里走

### 为什么先不直接给用户看

- 工作日志天然带有内部试探、候选决策和未定项
- 直接暴露给用户，容易让产品变成“系统自言自语的调试器”
- 更适合先作为开发、验收、调参和回放工具

---

## <a id="wl-10"></a>WL-10 风险与取舍

### 这个想法的优点

- 明显增强连续教学感
- 让“下一步建议”不再只是随机附加项
- 让首轮报告和多轮追问统一到一条教学主线上
- 为后续更强的 agent 化行为打基础

### 这个想法的主要风险

- 状态过多，系统可能开始“维护表格”而不是“教学生”
- 学生状态表如果过于自信，会制造伪理解判断
- 如果计划更新过于频繁，会让行为看起来飘忽不定

### 建议的取舍

- 保留三张表，但都做轻量版
- 优先让它们影响“下一轮怎么讲”，不要强行影响所有文案细节
- 学生状态表只做教学启发，不做强断言

我的判断是：

这个方向是对的，但要坚持“轻量、保守、可回退”。

不要把它做成复杂 agent orchestration；更像老师的课堂笔记、备课计划和对学生理解程度的粗粒度估计。

## <a id="wl-11"></a>WL-11 分阶段推进计划

### Phase 0

状态：partially_done

当前已经有：

- 首轮报告走 M6
- prompt 有初步 `teacher_memory`
- prompt 有初步 `teaching_plan`

### Phase 1

状态：proposed

目标：引入最小可用教学计划表

- 基于 M4 骨架生成首版计划
- 每轮回答参考当前计划
- 每轮结束后更新计划项状态

### Phase 2

状态：proposed

目标：引入最小可用学生学习状态表

- 对核心主题维护粗粒度理解状态
- 允许用用户信号触发 `needs_reinforcement`
- 让下一轮回答能根据该状态调节讲法

### Phase 3

状态：proposed

目标：引入教师 Working Log

- 记录本轮目标、理由、下一步过渡
- 与教学计划和学生状态关联
- 作为内部调试和回放信息

### Phase 4

状态：proposed

目标：引入流式内部日志

- 把状态选择和更新过程做成可观测事件流
- 优先服务调试和验收，不急于直接展示给终端用户

### Phase 5

状态：optional

目标：进一步 agent 化

- 允许“回答生成”与“状态更新”分成两个轻量步骤
- 允许在不调用外部工具的前提下，先规划再作答

注意：

这一阶段是可选增强，不应先于 Phase 1-4。

---

## 建议结论

后续 Agent 如果要继续推进“老师感”，优先顺序建议是：

1. 先落地教学计划表
2. 再落地学生学习状态表
3. 再落地教师 Working Log
4. 最后再做流式内部日志与更强 agent 化

原因很简单：

- 没有计划，就没有“主动带路”
- 没有学生状态，就没有“因材施教”的感觉
- 没有工作日志，就难以调试老师为什么这样教

但这三者都应保持轻量，不要反客为主。
