PRD：Repo Tutor
1. 产品概述
1.1 产品名称

Repo Tutor

1.2 产品定位

一个面向初学者的代码仓库教学 Agent。
它不是代码生成器，也不是普通问答机器人，而是一个会主动带路的源码学习引导系统。

1.3 产品本质

这个产品有两层定位：

第一层是教学产品壳子：

帮用户理解陌生代码仓库
帮用户建立工程认知
帮用户沿着主流程逐步深入

第二层是后续实验载体：

为未来验证 Virtual Context / Working Context Manager 预留接口
但第一版不把实验能力作为对外主卖点
2. 产品背景：为什么要做

对初学者来说，阅读真实工程的最大困难通常不是“某一行代码看不懂”，而是：

不知道什么东西值得先看
不知道什么叫入口、模块、分层、依赖、主流程
不知道一个仓库到底是“怎么动起来的”
看不出哪些是项目自己写的，哪些是标准库或第三方库
看了很多局部代码，仍然没有整体认知

也就是说，用户缺的不是更多代码解释，而是：

一套看工程的认知框架 + 一条能被带着走的理解路径

因此，这个产品的目标不是替用户写代码，而是替用户完成三件事：

建立工程认知
识别主流程和数据流
形成持续深入源码的能力
3. 目标用户：给谁用
3.1 核心用户

初学开源项目的大学生

3.2 用户特征
学过基础编程
能看懂简单代码
会写小项目，但不会看真实工程
面对陌生仓库容易迷路
缺少源码阅读方法
缺少工程认知框架
3.3 典型用户状态

用户常见的真实困惑不是：

“这段 if/for 我看不懂”

而是：

“我根本不知道这个工程里哪些东西重要”
“我不知道什么是入口”
“我不知道数据流是什么”
“我不知道模块为什么这样分”
“我不知道 import 进来的到底是谁家的东西”
“我不知道该先抓住整体还是细节”
4. 需求目标：解决什么问题
4.1 核心问题

Repo Tutor 要解决的是：

用户面对一个陌生仓库时，没有工程认知框架，不知道该看见什么、先理解什么、如何把模块和流程拼成一个整体。

4.2 目标拆解

产品第一版需要完成以下闭环：

目标 1：建立最小工程认知

让用户先知道：

什么是入口
什么是主流程
什么是模块
什么是分层
什么是依赖
什么是项目内部代码 / 标准库 / 第三方库
什么是核心路径 / 支撑路径
目标 2：建立仓库整体结构理解

让用户知道：

仓库主要由哪些部分组成
核心目录和核心模块是什么
主入口和关键流程在哪里
目标 3：生成可执行的阅读路径

让用户知道：

先看哪里
为什么先看这个
接下来该看什么
目标 4：以数据流和调用链为主线教学

让用户知道：

输入从哪里进入系统
数据如何流转
模块如何拼接
各层在流程中扮演什么角色
目标 5：支持围绕当前仓库持续深入

让用户可以围绕当前仓库进行多轮追问，并且 agent 不丢当前语境。

5. 一句话定义

Repo Tutor 是一个以工程认知、数据流和主流程为主线的引导式源码教学 Agent，帮助初学者从“看不懂陌生仓库”变成“知道先看哪、系统怎么运转、模块怎么拼起来”。

6. 输入范围：接收什么内容
6.1 第一版支持
本地仓库路径
GitHub 公共仓库 URL
6.2 第一版不支持
私有仓库认证接入
PDF / 课件 / 设计文档混合输入
多仓库联合教学
在线运行代码环境接入
7. 输出范围：给用户什么结果

第一版必须输出以下内容。

7.1 仓库结构总览
仓库整体由哪些关键部分组成
哪些目录/模块最重要
入口文件在哪里
7.2 阅读路径
第一眼看什么
第二步看什么
每一步为什么这样安排
7.3 模块讲解
核心模块负责什么
模块之间怎么协作
一个模块在整体中的角色是什么
7.4 数据流 / 主流程讲解
数据从哪里来
经过哪些层
每一步做了什么
最后去向哪里
7.5 依赖来源解释
某个 import 是项目内部模块、标准库还是第三方库
它在当前流程里扮演什么角色
当前是否值得展开看
7.6 多轮问答

支持围绕当前仓库持续提问，例如：

启动流程怎么走
登录流程怎么走
这个模块为什么存在
这里为什么要调这个依赖
这个 import 是不是项目自己写的
8. 教学理念：怎么教
8.1 引导，而不是指导

第一版不是单纯“你问我答”的指导器，而是引导式教学系统。

它应先帮助用户建立观察框架，再把这个框架映射到当前仓库，最后沿主流程逐步深入。

8.2 强主动

系统默认应主动带路，而不是停下来等待用户想问题。

每轮讲解后，系统默认要：

总结本轮学到了什么
指出当前还缺什么关键认知
给出下一步建议
8.3 可打断

虽然系统强主动，但用户始终可以中途打断，重定方向或深度，例如：

先不要讲模块，讲数据流
我只想看启动流程
不用解释基础概念
讲深一点
8.4 先骨架，后细节

默认讲解顺序是：

先整体
再流程
再模块关系
最后才进入局部实现细节
8.5 粗粒度推进

每轮默认只推进少量关键点：

2～4 个核心认知点
1 条主流程线索
1 个下一步建议

不追求一轮讲全。

8.6 连续讲解优先

主流程里不频繁插入“猜一猜”“你先回答”这类微任务。
如果要做理解检查，应放在阶段性总结里，而且默认可跳过。

8.7 自然语言解释优先

第一版主输出以直接自然语言讲解为主，不以抽象 txt 图、ASCII 图为主要承载形式。图示最多作为辅助。

9. 核心产品体验

Repo Tutor 应让用户逐渐形成这种体验：

我知道这一轮主要在学什么
我知道为什么先学这个
我开始能区分工程中的不同角色
我开始能看见主流程，而不是只看见文件
我开始知道模块是怎么拼起来的
我被带着进入一个略有挑战但能跟上的节奏
读源码不再只是噩梦，而开始有“听懂了”的成就感
10. 功能需求
10.1 Must Have
F1. 仓库接入

支持：

读取本地仓库路径
拉取或读取 GitHub 公共仓库 URL

系统能够建立最基本的仓库内容访问能力。

F2. 文件树扫描与关键对象识别

系统能够：

扫描目录结构
识别候选入口文件
识别关键目录与核心模块
识别关键配置文件
识别常见依赖声明位置

输出必须是教学可用的结构，不是简单原样 dump 整棵树。

F3. 工程认知引导

在正式讲解仓库前，系统必须先帮助用户建立最小观察框架，例如：

看仓库先看什么
入口 / 模块 / 分层 / 依赖 / 主流程是什么意思
当前仓库里这些东西分别如何体现

这不是脱离上下文的通识课，而是服务于当前仓库。

F4. 仓库结构总览生成

系统能够生成适合初学者理解的仓库整体说明，包括：

核心模块划分
关键目录说明
主入口位置
大体结构关系
F5. 阅读路径生成

系统能够输出面向初学者的阅读顺序，并为每一步说明：

为什么先看它
这一阶段是为了建立什么认知
暂时可以先忽略哪些内容
F6. 数据流 / 主流程讲解

系统必须能围绕一条主流程讲清：

输入从哪里进入
经过哪些函数/模块/层
每一步做了什么处理
输出如何产生

这是第一版的核心能力之一。

F7. 模块关系讲解

系统能够解释：

模块如何拼接
上层如何调用下层
模块在整体中的角色
哪些模块是主路径，哪些是支撑层
F8. 依赖来源识别与讲解

系统必须主动区分并解释：

项目内部模块
Python 标准库
第三方库/框架

并说明：

它在当前流程中的作用
当前是否值得展开
F9. 分层视角解释

系统应尝试给当前仓库建立教学式分层视图，哪怕是启发式的，不要求绝对严格。

例如帮助用户理解：

入口层
路由/控制层
业务逻辑层
数据访问层
工具/配置层

这里的目的不是追求架构学术严谨，而是帮助新手建立工程感。

F10. 多轮问答

用户可以围绕当前仓库持续提问。系统应保持：

当前讨论主题
当前聚焦模块
当前主流程语境
已经解释过的关键点

避免每一轮都重新开始。

F11. 学习目标切换

用户可以显式指定当前学习目标，例如：

只想理解启动流程
先看登录流程
只看数据库层
先别管整体，先讲 import 来源

系统要围绕目标重新组织讲解。

F12. 学习状态维护

系统至少维护以下状态：

当前仓库
当前学习目标
当前讲解阶段
当前聚焦模块
已讲解模块/文件
当前主流程上下文
当前分层理解结果
当前依赖归类结果
10.2 Should Have
S1. 文件/模块摘要缓存

对文件和模块讲解结果进行缓存，减少重复解析和重复总结。

S2. 主流程候选路径提示

例如：

如果想理解启动流程，看 A → B → C
如果想理解登录流程，看 X → Y → Z
S3. 实验日志记录

记录：

当前问题
召回了哪些上下文单元
使用了哪些摘要/原文
最终组装了哪些上下文

这为你后续做 virtual context 实验提供基础。

S4. 可切换上下文组装接口

虽然第一版不主打实验，但内部结构应支持后续替换不同上下文策略。

S5. 阶段性总结

在一个阶段结束时，系统能输出：

当前已经建立了哪些理解
还缺什么
下一阶段建议看什么

必要时可附带轻量自查问题，但默认不阻塞继续学习。

10.3 Won’t Have

第一版明确不做：

自动运行仓库代码
自动修改代码
自动生成 PR / 提交代码
私有仓库接入
PDF/文档/课件混合理解
多仓库联合教学
通用 Agent OS
真正意义上的无限上下文实现
复杂可视化架构图作为主交互
高频互动式小测问答主流程
11. 业务流程：用户怎么用
11.1 进入

用户输入：

本地仓库路径，或
GitHub 公共仓库 URL
11.2 初始化解析

系统：

扫描文件树
识别关键文件和关键模块
提取候选入口
做初步依赖归类
做初步流程/分层判断
11.3 首轮引导讲解

系统先不直接平铺结构，而是先说：

看一个仓库应该先抓哪些东西
当前仓库里这些东西分别在哪里
第一阶段最值得看的是什么
11.4 主动推进

系统默认继续带路，例如：

先看入口
再跟主流程
再识别核心模块
再区分主路径和支撑层
11.5 用户可打断改向

用户可以随时说：

先讲数据流
只看 auth 模块
不解释基础概念
讲深一点

系统根据新目标重组当前讲解路径。

11.6 阶段总结

一个主题阶段结束后，系统总结：

你已经掌握了什么
还缺什么关键认知
接下来最推荐看什么
11.7 持续循环

直到用户结束会话或切换到新的仓库/目标。

12. 页面 / 交互说明

第一版建议采用CLI 或极简对话式界面。不做重 UI。

12.1 核心输入
仓库路径输入框 / 命令
GitHub URL 输入框 / 命令
对话输入框
12.2 核心状态显示

界面应能显示最少这些状态：

当前仓库名
当前学习目标
当前讲解阶段
当前聚焦模块
下一步建议
12.3 首轮输出形式

首轮输出默认包含：

现在看仓库要先抓什么
这些东西在当前仓库里的映射
当前最推荐的第一步
接下来可继续看的方向
12.4 每轮输出形式

每轮默认应包含：

本轮重点
对当前重点的直接解释
它和整体的关系
本轮总结
下一步建议
12.5 主动建议形式

由于系统采用强主动设计，每轮末尾都要有明确推进，例如：

下一步我建议跟一遍启动流程
现在你已经知道入口了，接下来最关键的是看数据怎么进入业务层
你已经区分出内部模块和第三方依赖了，下一步建议只看项目自己写的核心链路
13. 范围边界
13.1 本次做什么
做一个能教用户读仓库的单 agent 系统
重点做工程认知搭建、数据流讲解、主流程引导
支持目标切换和多轮教学
内部预留后续 virtual context 实验接口
13.2 本次不做什么
不做通用 agent 平台
不做多 agent 编排系统
不做代码执行和自动修复
不做复杂知识融合系统
不做完整的实验论文平台

一句话边界：

只做“理解仓库”的教学 Agent，不做“操作仓库”的工程 Agent。

14. 验收标准
14.1 功能验收

系统应至少能在 3 个公开仓库或本地仓库上完成以下能力：

成功接入仓库
生成结构总览
给出阅读路径
讲清一条主流程
区分核心依赖来源
支持多轮围绕同一目标的追问
14.2 教学效果验收

系统应能够回答并讲清以下类型的问题：

入口文件在哪里，为什么它算入口
数据从哪里进入，怎么流转
核心模块有哪些，各自做什么
某个 import 是内部模块、标准库还是第三方库
当前仓库大体是如何分层的
启动流程 / 登录流程 / 某主路径如何走通
14.3 体验验收

用户应能明显感受到：

系统在主动带路
单轮信息量可消化
不会被频繁打断做小题
可以随时改向
默认讲解是新手能接住的
14.4 状态验收

在连续多轮问答中，系统应能保持：

当前聚焦主题一致
当前主流程语境一致
已解释内容不反复从零开始
14.5 架构验收

内部实现上，应至少为以下模块预留清晰接口：

Context Source
Context Retriever
Context Compressor
Context Assembler
Context Policy
State Tracker

即使第一版不把它们全做强，也必须避免把上下文管理硬编码死在一个 prompt 里。

16. 产品成功标准

如果这个产品第一版是成功的，应该满足下面这句话：

一个会基础编程但不会看工程的大学生，在使用 Repo Tutor 后，能够在陌生仓库中建立最小工程认知，知道入口、主流程、模块关系、依赖来源和阅读顺序，并能在 agent 主动引导下持续深入，而不是在文件堆里迷失。

17. 给架构 agent 的一句话任务定义

基于本 PRD 设计一个单 agent 的教学系统架构，核心围绕“工程认知引导 + 数据流主线讲解 + 强主动多轮教学 + 可插拔上下文管理层”，严格控制边界，禁止发散为通用 agent OS 或代码执行平台。
