# PRD v2: Repo Tutor

版本：v2.0  
来源：基于 `docs/PRD_v1.txt` 的产品与架构评审重写  
目标读者：产品、架构、研发、测试

## 0. v1 架构评审结论

### 0.1 是否能理解 v1 的想法

能理解。

Repo Tutor 的核心意图是：为不会阅读真实工程的初学者提供一个“主动带路”的源码学习 Agent。它不替用户写代码，也不运行或修改仓库，而是围绕仓库入口、主流程、数据流、模块关系、依赖来源和阅读路径，帮助用户建立最小工程认知，并支持围绕当前仓库持续追问。

v1 的产品方向是成立的，用户痛点也清楚：初学者不是缺少单行代码解释，而是缺少阅读真实工程的观察框架和推进路径。

### 0.2 v1 不够适合直接交给架构师的地方

1. **语言范围不明确**  
   v1 写了 GitHub 公共仓库和本地仓库，但功能需求里又明确出现 Python 标准库、Python 依赖识别。架构上必须明确 MVP 是否只支持 Python，否则入口识别、依赖分类、主流程分析都无法设计边界。

2. **“讲清主流程”风险过大**  
   对任意仓库自动讲清数据流和调用链，静态分析难度很高。v2 必须限定：第一版以静态启发式分析为主，输出要标注证据、置信度和不确定项，不能伪装成完全准确。

3. **缺少输出契约**  
   v1 描述了要输出结构总览、阅读路径、模块讲解等，但没有规定每类输出必须包含哪些字段、是否需要引用文件证据、如何表达推断。架构师需要稳定的数据结构和渲染契约。

4. **上下文管理只提了方向，没有状态模型**  
   v1 说要预留 Virtual Context / Working Context Manager，但没有定义学习状态、仓库索引、摘要缓存、召回策略、日志之间的关系。v2 需要把这些拆成可替换模块。

5. **验收标准偏感受，缺少可测条件**  
   “系统在主动带路”“新手能接住”是体验目标，但测试需要更具体的输入样本、输出要求、失败兜底、性能和准确性指标。

6. **缺少安全与仓库访问边界**  
   读取本地仓库和 GitHub URL 必须明确：不执行代码、不上传敏感文件、忽略大文件和 vendor 目录、不处理私有认证、不修改仓库。

7. **单 Agent 与内部模块边界容易混淆**  
   v1 要做单 Agent 系统，同时又要预留多个上下文管理组件。v2 明确：对外是单 Agent 体验，内部可以是模块化服务或组件，但不是多 Agent 编排平台。

## 1. 产品概述

### 1.1 产品名称

Repo Tutor

### 1.2 一句话定义

Repo Tutor 是一个面向初学者的源码仓库教学 Agent，通过静态分析仓库结构、入口、依赖、主流程候选路径和模块关系，主动引导用户建立工程认知，并给出可持续深入的阅读路径。

### 1.3 产品定位

Repo Tutor 不是代码生成器、代码执行器、自动修复工具或通用 Agent OS。它是一个“理解仓库”的教学型 Agent。

对外定位：

- 帮初学者看懂陌生代码仓库。
- 帮初学者建立入口、模块、分层、依赖、主流程、数据流等工程认知。
- 通过强主动的教学节奏带用户逐步深入，而不是等待用户自己提出正确问题。

内部定位：

- 第一版保留可插拔上下文管理接口。
- 第一版不把 Virtual Context / Working Context Manager 作为对外主卖点。
- 第一版只要求上下文管理边界清楚，不要求完成复杂实验能力。

## 2. 用户与场景

### 2.1 核心用户

学过基础编程、能看懂简单代码、但缺少真实工程阅读经验的大学生或初级开发者。

### 2.2 用户能力假设

- 会打开一个本地仓库或 GitHub 公共仓库。
- 能理解基础函数、类、条件分支、循环和 import。
- 不熟悉大型项目的目录组织、入口识别、分层设计、依赖来源和主流程阅读方法。

### 2.3 核心用户问题

- 不知道先看哪些文件。
- 不知道入口文件在哪里。
- 不知道模块为什么这样分。
- 不知道 import 来自项目内部、标准库还是第三方库。
- 看了很多局部代码，仍然拼不出系统如何运转。
- 多轮提问后，普通问答 Agent 容易丢失当前仓库和当前学习目标。

### 2.4 典型使用场景

1. 用户输入一个本地 Python 仓库路径，希望系统告诉他先看哪里。
2. 用户输入一个 GitHub 公共 Python 仓库 URL，希望理解启动流程。
3. 用户已经看完入口讲解，继续追问某个模块为什么存在。
4. 用户中途切换目标，从“看整体结构”改成“只看依赖来源”。
5. 用户围绕一个主流程连续追问，期望系统不重复从零开始。

## 3. MVP 范围

### 3.1 第一版明确支持

- 本地仓库路径输入。
- GitHub 公共仓库 URL 输入。
- Python 仓库的静态分析与教学讲解。
- 常见 Python 依赖声明识别：`pyproject.toml`、`requirements.txt`、`setup.py`、`setup.cfg`、`Pipfile`。
- 常见 Python 入口候选识别：`__main__.py`、`main.py`、`app.py`、`manage.py`、CLI scripts、README 中的运行命令、框架约定入口。
- 围绕当前仓库的多轮问答和学习状态维护。

### 3.2 第一版有限支持

对非 Python 仓库，第一版只能输出文件树级别的结构总览和阅读建议，必须标注“语言暂不完整支持”。不承诺准确的依赖来源、调用链或数据流讲解。

### 3.3 第一版不支持

- 私有仓库认证接入。
- 在线运行仓库代码。
- 自动修改代码。
- 自动生成 PR 或提交代码。
- 多仓库联合教学。
- PDF、课件、设计文档混合输入。
- 通用 Agent OS。
- 多 Agent 编排平台。
- 完整跨语言调用链分析。
- 复杂可视化架构图作为主要交互。

## 4. 产品目标

### 4.1 Primary Goal

帮助初学者在陌生 Python 仓库中建立最小工程认知，并知道入口、主流程、模块关系、依赖来源和下一步阅读路径。

### 4.2 Secondary Goals

1. 让用户知道当前仓库最值得先看的文件和模块。
2. 让用户理解当前仓库的主要目录和模块分工。
3. 让用户沿一条候选主流程理解输入、处理和输出。
4. 让用户能区分内部模块、标准库和第三方库。
5. 让系统在多轮对话中保持当前仓库、目标、阶段和上下文。
6. 为后续可替换上下文策略预留模块边界与日志。

### 4.3 Non-Goals

- 不替用户写业务代码。
- 不评价代码质量或给出重构方案，除非用户明确切换到代码评审场景。
- 不保证完整、精确、跨动态运行时的调用链。
- 不在第一版实现私有仓库访问。
- 不把实验性的上下文管理能力暴露成主功能。

## 5. 核心体验原则

### 5.1 主动带路

系统默认主动推进学习路径，每轮输出都要给出下一步建议。用户无需一开始就知道该问什么。

### 5.2 可打断、可改向

用户可以随时改变学习目标，例如“只看启动流程”“先讲 import 来源”“不要解释基础概念”。系统必须基于已有仓库分析结果重组讲解，而不是重新开始。

### 5.3 先骨架，后细节

默认讲解顺序为：仓库整体结构 -> 入口与主流程候选 -> 模块关系 -> 局部实现细节。

### 5.4 证据优先

系统对仓库的判断必须尽量引用文件路径、符号、配置或 README 线索。无法确认时必须标注为推断或未知，不能编造确定结论。

### 5.5 小步推进

每轮默认只推进少量关键点，避免一次性讲完整个仓库。建议每轮包含 2 到 4 个核心认知点、1 条流程线索、1 个下一步建议。

## 6. 核心功能需求

### F1. 仓库接入

优先级：P0

系统必须支持：

- 接收本地仓库路径。
- 接收 GitHub 公共仓库 URL。
- 建立只读仓库访问。
- 识别仓库名称、根目录、主要语言、文件数量和体量。

约束：

- 不执行仓库中的任何代码。
- 不修改仓库文件。
- 不读取 `.env`、密钥文件、凭据文件内容。
- GitHub URL 只支持公开仓库。

验收：

- 给定一个本地 Python 仓库，系统能返回仓库基本信息。
- 给定一个 GitHub 公共 Python 仓库 URL，系统能完成拉取或读取，并进入分析阶段。
- 给定私有仓库 URL，系统能明确提示第一版不支持私有认证。

### F2. 文件树扫描与过滤

优先级：P0

系统必须扫描仓库文件树，并过滤不适合教学分析的内容。

必须识别并默认忽略：

- `.git/`
- `node_modules/`
- `.venv/`
- `venv/`
- `dist/`
- `build/`
- `__pycache__/`
- 大型二进制文件
- 压缩包
- 图片、音视频等非源码资产

输出要求：

- 不是原样 dump 整棵树。
- 必须提炼出教学可用的关键目录、候选模块和配置文件。
- 必须保留完整文件索引用于后续检索。

### F3. 语言与项目类型识别

优先级：P0

系统必须识别：

- 主语言。
- Python 项目类型候选，例如 CLI、Web App、Library、Script Collection、Notebook Project。
- 关键配置文件。
- README 中与运行、安装、入口相关的线索。

当语言或项目类型无法确认时：

- 标注 `UNKNOWN`。
- 只输出低置信度结构总览。
- 不强行生成确定主流程。

### F4. 入口候选识别

优先级：P0

系统必须识别 Python 仓库的候选入口，并按置信度排序。

入口线索包括：

- `__main__.py`
- `main.py`
- `app.py`
- `manage.py`
- `if __name__ == "__main__"`
- `pyproject.toml` scripts
- `setup.py` entry_points
- README 运行命令
- 常见框架约定入口

输出必须包含：

- 入口候选文件。
- 入口判断依据。
- 置信度：`HIGH`、`MEDIUM`、`LOW`。
- 若有多个入口，说明推荐先看的入口和原因。

### F5. 依赖来源识别

优先级：P0

系统必须区分：

- 项目内部模块。
- Python 标准库。
- 第三方库。
- 未知来源。

输出必须说明：

- import 名称。
- 来源类型。
- 相关文件。
- 在当前流程中的作用。
- 当前是否值得展开阅读。

约束：

- 标准库判断必须基于当前 Python 版本的标准库集合或等价机制。
- 第三方库判断必须优先来自项目依赖声明，其次来自 import 启发式判断。
- 无法判断时必须标注 `UNKNOWN`。

### F6. 仓库结构总览

优先级：P0

系统必须生成面向初学者的结构总览，包括：

- 仓库由哪些关键部分组成。
- 哪些目录或模块最值得先看。
- 哪些目录可以暂时跳过。
- 主入口候选在哪里。
- 当前仓库可能属于什么项目类型。

输出风格：

- 自然语言优先。
- 可附轻量列表。
- 不以复杂图示作为主输出。

### F7. 阅读路径生成

优先级：P0

系统必须为初学者生成阅读路径。

每一步必须包含：

- 建议看的文件或模块。
- 为什么先看它。
- 这一阶段建立什么认知。
- 暂时可以忽略什么。
- 下一步可能看哪里。

阅读路径必须可被用户打断和改向。

### F8. 主流程 / 数据流讲解

优先级：P0

系统必须围绕一条候选主流程讲解：

- 输入从哪里进入。
- 经过哪些函数、类、模块或层。
- 每一步大致做什么。
- 输出如何产生。
- 哪些判断来自直接证据，哪些是推断。

约束：

- 第一版只承诺静态启发式流程讲解。
- 不承诺覆盖动态注册、反射、装饰器副作用、框架运行时注入导致的完整真实调用链。
- 当流程无法确认时，输出“候选流程”而不是“真实流程”。

### F9. 模块关系讲解

优先级：P0

系统必须解释核心模块之间的协作关系：

- 模块职责。
- 上游和下游关系。
- 是否属于主路径。
- 是否属于支撑层。
- 与当前学习目标的关系。

### F10. 工程认知引导

优先级：P0

系统在正式讲仓库前必须先建立最小观察框架。

框架包括：

- 入口是什么。
- 主流程是什么。
- 模块是什么。
- 分层是什么。
- 依赖是什么。
- 内部模块、标准库、第三方库如何区分。
- 当前仓库中这些概念分别映射到哪里。

约束：

- 不做脱离当前仓库的长篇通识课。
- 所有概念解释都必须服务于当前仓库阅读。

### F11. 多轮问答与学习状态维护

优先级：P0

系统必须维护当前学习状态，并在多轮问答中使用。

状态至少包括：

- 当前仓库。
- 当前学习目标。
- 当前讲解阶段。
- 当前聚焦模块。
- 当前主流程上下文。
- 已讲解文件。
- 已讲解模块。
- 已确认入口候选。
- 已归类依赖。
- 已生成阅读路径。
- 最近一轮下一步建议。

系统必须避免：

- 每轮从零重新介绍仓库。
- 忘记用户刚刚切换的学习目标。
- 对已讲过的内容进行无意义重复。

### F12. 学习目标切换

优先级：P1

用户可以显式切换目标：

- 只看启动流程。
- 只看登录流程。
- 只看某个模块。
- 只看依赖来源。
- 讲浅一点。
- 讲深一点。
- 不解释基础概念。

系统必须基于新目标更新学习状态，并重组下一轮输出。

### F13. 阶段性总结

优先级：P1

在一个主题阶段结束时，系统应输出：

- 本阶段建立了哪些理解。
- 当前还缺什么关键认知。
- 下一阶段建议看什么。
- 可选的轻量自查问题。

自查问题默认不阻塞继续学习。

### F14. 摘要缓存

优先级：P1

系统应缓存：

- 文件摘要。
- 模块摘要。
- 入口候选分析结果。
- 依赖归类结果。
- 主流程候选结果。

缓存必须能基于仓库版本或文件内容变化失效。

### F15. 实验日志

优先级：P2

系统应记录内部实验日志，用于后续上下文管理实验。

日志至少包括：

- 用户问题。
- 当前学习状态。
- 召回的上下文单元。
- 使用的摘要与原文证据。
- 最终组装的上下文。
- 输出中引用的证据。

约束：

- 日志默认不作为用户主界面内容。
- 日志不得包含密钥文件内容。

## 7. 输出契约

### 7.1 FirstRoundReport

首轮输出必须包含：

| 字段 | 说明 |
|---|---|
| repoSummary | 仓库一句话概览 |
| languageProfile | 主语言与项目类型候选 |
| observationFrame | 看当前仓库先抓哪些东西 |
| keyDirectories | 关键目录及原因 |
| entryCandidates | 入口候选及置信度 |
| recommendedFirstStep | 推荐第一步 |
| nextOptions | 后续可选方向 |
| uncertainty | 目前不确定或需要后续确认的点 |

### 7.2 ReadingPath

阅读路径每一步必须包含：

| 字段 | 说明 |
|---|---|
| stepId | 顺序编号 |
| target | 文件、目录或模块 |
| purpose | 为什么看它 |
| learningOutcome | 看完应建立什么认知 |
| skipForNow | 暂时不用看的内容 |
| evidence | 路径、配置或 README 线索 |
| nextStep | 下一步建议 |

### 7.3 FlowWalkthrough

主流程讲解必须包含：

| 字段 | 说明 |
|---|---|
| flowName | 流程名称 |
| confidence | `HIGH` / `MEDIUM` / `LOW` |
| entryPoint | 入口文件或函数 |
| steps | 候选流程步骤 |
| dataMovement | 数据如何流转 |
| moduleRoles | 相关模块角色 |
| evidence | 支撑判断的文件或符号 |
| unknowns | 无法确认的部分 |

### 7.4 DependencyExplanation

依赖说明必须包含：

| 字段 | 说明 |
|---|---|
| importName | import 名称 |
| sourceType | `INTERNAL` / `STDLIB` / `THIRD_PARTY` / `UNKNOWN` |
| usedBy | 使用它的文件或模块 |
| roleInFlow | 在当前流程中的作用 |
| shouldExpandNow | 当前是否值得展开 |
| evidence | 判断依据 |

### 7.5 LearningState

系统内部必须维护：

| 字段 | 说明 |
|---|---|
| repoId | 仓库唯一标识 |
| repoSource | 本地路径或 GitHub URL |
| repoVersion | commit hash 或本地快照指纹 |
| learningGoal | 当前学习目标 |
| teachingStage | 当前讲解阶段 |
| focusModule | 当前聚焦模块 |
| explainedFiles | 已讲解文件集合 |
| explainedModules | 已讲解模块集合 |
| selectedFlow | 当前主流程 |
| dependencyMapVersion | 依赖图版本 |
| repositoryIndexVersion | 仓库索引版本 |
| nextSuggestion | 下一步建议 |

## 8. 系统边界与架构约束

### 8.1 对外边界

Repo Tutor 对外表现为单 Agent 对话体验。用户不需要知道内部有多少模块，也不需要选择分析器、检索器或上下文策略。

### 8.2 内部模块边界

内部实现应至少拆分为以下职责模块：

1. **RepositoryAccess**  
   负责只读接入本地仓库或 GitHub 公共仓库。

2. **RepositoryIndexer**  
   负责文件树扫描、过滤、语言识别、关键文件识别和索引生成。

3. **PythonStaticAnalyzer**  
   负责 Python 入口候选、import 来源、模块关系和候选流程线索分析。

4. **LearningStateTracker**  
   负责维护当前仓库、学习目标、讲解阶段、焦点模块和已解释内容。

5. **ContextManager**  
   负责上下文召回、摘要压缩、上下文组装和预算控制。

6. **TeachingPlanner**  
   负责根据用户目标和当前状态决定下一轮讲解重点。

7. **TeachingResponder**  
   负责生成面向初学者的自然语言讲解。

8. **TraceLogger**  
   负责记录分析证据、上下文召回和输出依据，为后续实验提供数据。

### 8.3 依赖方向

推荐依赖方向：

```text
RepositoryAccess -> RepositoryIndexer -> PythonStaticAnalyzer
RepositoryIndexer -> ContextManager
PythonStaticAnalyzer -> ContextManager
LearningStateTracker -> TeachingPlanner
ContextManager -> TeachingPlanner
TeachingPlanner -> TeachingResponder
TeachingResponder -> LearningStateTracker
TeachingPlanner -> TraceLogger
ContextManager -> TraceLogger
PythonStaticAnalyzer -> TraceLogger
```

约束：

- TeachingResponder 不得直接读取仓库文件，必须通过 ContextManager 获取证据。
- PythonStaticAnalyzer 不得生成面向用户的教学文案。
- RepositoryAccess 不得执行仓库代码。
- ContextManager 不得修改分析结论，只负责选择和压缩上下文。
- LearningStateTracker 不得存储仓库文件全文。

### 8.4 上下文管理预留接口

ContextManager 内部建议拆分为：

- Context Source：提供仓库文件、摘要、分析结果、对话历史、学习状态。
- Context Retriever：根据当前问题和学习目标召回上下文。
- Context Compressor：压缩文件摘要或历史讲解。
- Context Assembler：组装最终上下文。
- Context Policy：控制预算、优先级和证据选择策略。
- State Tracker：同步学习状态。

v1 架构验收重点不是复杂策略效果，而是这些职责不能硬编码在一个 prompt 中。

## 9. 安全与隐私约束

1. 系统不得执行仓库中的代码、脚本、测试或安装命令。
2. 系统不得修改、删除或格式化仓库文件。
3. 系统不得读取常见敏感文件内容，例如 `.env`、密钥、证书、token 文件。
4. 系统必须默认忽略大型二进制文件和依赖目录。
5. GitHub 第一版只支持公开仓库。
6. 用户提供本地路径时，系统只读访问该路径下的仓库内容。
7. 输出中不得泄露疑似密钥或凭据信息。

## 10. 错误与兜底

| 场景 | 系统行为 |
|---|---|
| 本地路径不存在 | 明确提示路径不可访问，不进入分析 |
| GitHub URL 非公开或无法访问 | 提示第一版只支持公开仓库 |
| 仓库过大 | 提示已启用过滤和采样，并说明可能影响完整性 |
| 非 Python 仓库 | 输出有限结构总览，标注语言暂不完整支持 |
| 找不到入口 | 给出候选阅读路径，但标注入口未知 |
| 多个入口 | 按置信度排序，并解释推荐先看哪个 |
| 依赖无法归类 | 标注 `UNKNOWN`，不强行解释 |
| 主流程无法确认 | 输出候选流程或建议先看结构总览 |
| 用户切换目标 | 更新学习状态，并从新目标继续 |

## 11. 非功能需求

### 11.1 性能

第一版目标：

- 小型仓库：少于 500 个源码文件，首轮分析在 30 秒内开始输出。
- 中型仓库：500 到 3000 个源码文件，首轮分析在 90 秒内开始输出。
- 大型仓库：超过 3000 个源码文件，必须启用过滤、摘要和分阶段分析。

### 11.2 准确性

- 所有确定性判断必须有证据。
- 所有推断必须标注为推断。
- 无法判断必须输出未知，而不是编造。

### 11.3 可维护性

- 分析模块、教学模块、上下文模块必须分离。
- 输出契约必须稳定。
- 新增语言支持时，不应改动 TeachingResponder 的核心教学逻辑。

### 11.4 可测试性

- RepositoryIndexer、PythonStaticAnalyzer、LearningStateTracker、ContextManager 必须可独立测试。
- 至少准备 3 个固定样例仓库作为回归测试对象。

## 12. 验收标准

### 12.1 功能验收

系统必须在至少 3 个 Python 仓库上完成：

1. 成功接入仓库。
2. 生成仓库结构总览。
3. 给出阅读路径。
4. 识别至少一个入口候选。
5. 区分内部模块、标准库和第三方库。
6. 讲清一条候选主流程。
7. 支持围绕同一仓库的多轮追问。
8. 支持用户中途切换学习目标。

建议样例仓库类型：

- 一个小型 CLI 项目。
- 一个小型 Web 项目。
- 一个 Python library/package 项目。

### 12.2 教学验收

对每个样例仓库，系统必须能回答：

- 入口文件在哪里，为什么它可能是入口。
- 当前仓库主要由哪些模块组成。
- 哪些文件值得先看，哪些可以暂时跳过。
- 一条候选主流程怎么走。
- 某个 import 是内部模块、标准库、第三方库还是未知。
- 当前讲解下一步建议是什么。

### 12.3 状态验收

连续 5 轮对话中，系统必须保持：

- 当前仓库一致。
- 当前学习目标一致，除非用户主动切换。
- 当前聚焦模块一致或能解释为什么切换。
- 已讲解内容不重复从零开始。

### 12.4 架构验收

必须满足：

- RepositoryAccess 不执行代码。
- RepositoryIndexer 可单独运行并输出索引。
- PythonStaticAnalyzer 可单独运行并输出入口、依赖、模块关系。
- ContextManager 可替换策略实现。
- TeachingResponder 不直接访问文件系统。
- TraceLogger 能记录召回证据和输出依据。

### 12.5 体验验收

用户在首轮输出中应能明确知道：

- 这个仓库大概是什么。
- 先看什么。
- 为什么先看这个。
- 下一步可以怎么继续。
- 当前有哪些不确定项。

## 13. 优先级

### P0

- 仓库接入。
- 文件树扫描与过滤。
- Python 项目类型识别。
- 入口候选识别。
- 依赖来源识别。
- 仓库结构总览。
- 阅读路径生成。
- 候选主流程讲解。
- 工程认知引导。
- 多轮问答状态维护。

### P1

- 学习目标切换。
- 阶段性总结。
- 摘要缓存。
- 分层视角解释。
- 主流程候选路径列表。

### P2

- 实验日志。
- 可替换 Context Policy。
- 更完整的 Context Compressor。
- 多种上下文组装策略对比。

### P3

- 更多语言支持。
- 私有仓库认证。
- 更复杂可视化。
- 文档混合输入。
- 多仓库联合教学。

## 14. 里程碑建议

### M1. 可读仓库

完成 RepositoryAccess、RepositoryIndexer、基础过滤和仓库概览。

### M2. 可识别关键对象

完成 Python 入口候选、依赖来源、核心目录和模块候选识别。

### M3. 可教学首轮

完成 FirstRoundReport、工程认知引导和阅读路径。

### M4. 可连续教学

完成 LearningState、目标切换、多轮追问和阶段总结。

### M5. 可实验扩展

完成 ContextManager 抽象、TraceLogger 和摘要缓存。

请基于本 PRD 设计一个对外单 Agent、内部模块化的 Repo Tutor 系统。第一版聚焦 Python 仓库的静态教学分析，围绕“工程认知引导 + 入口识别 + 依赖来源 + 候选主流程 + 多轮学习状态 + 可插拔上下文管理”落地，禁止扩展成代码执行平台、通用 Agent OS 或多 Agent 编排系统。
