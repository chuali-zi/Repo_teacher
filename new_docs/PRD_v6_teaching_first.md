## <a id="product"></a>PRODUCT — 产品定义与教学中心

### 产品名称
`Repo Tutor`

### 一句话定义
面向初中级开发者的**只读**源码教学 Agent。它以真实仓库为教学素材，帮助用户建立工程认知、理解系统结构与设计意图，并沿着精细的教学推进持续深入，而不是停留在保守解释和证据罗列。

### 产品类型
- 教学型 Agent，不是工程执行 Agent
- 单 Agent 体验，不是多 Agent 编排平台
- 第一版只读，不执行、不修改仓库

### 产品本质
Repo Tutor 的核心不是“回答仓库问题”，而是“带用户读懂仓库”。

系统应把源码当作教学素材而不是输出边界：

- 源码负责提供真实上下文、约束和落点
- 教学负责组织内容、建立认知、解释设计、安排推进
- 证据负责防止胡编，但不能吞掉教学正文

### 第一版教学主线
第一版围绕以下主线组织：

1. 帮用户先建立最小工程认知，而不是先堆细节
2. 在当前仓库里找出最值得先讲透的一个点
3. 用少量源码锚点把这个点讲清楚
4. 给出紧接当前点的下一教学细点
5. 让多轮对话像一堂持续推进的课，而不是每轮重新开始

### 产品成功标准
一个会基础编程但不会阅读真实工程的用户，在使用 Repo Tutor 后，能够：

- 用工程语言说清这个系统在解决什么问题、主要由哪些部分构成
- 理解一个关键设计点为什么这么设计，而不是只知道它在哪个文件
- 沿教学路径继续读源码，而不是在文件树里迷失
- 把学到的认知框架迁移到别的仓库，而不是只记住当前仓库的局部事实

### 资源分配原则
系统应明确优先把 LLM 预算用在教学内容上，而不是过度消耗在证据推理上。

优先项：

- 内容组织
- 单点深入解释
- 设计意图与职责拆分讲解
- 与整体关系说明
- 下一教学细点推进

辅助项：

- 证据整理
- 置信度与候选标注
- 不确定性提示

原则：只要证据足以支撑可靠教学，就不应继续扩张证据篇幅。

## <a id="users"></a>USERS — 用户与能力假设

### 目标用户
大学生、实习生、初级开发者、第一次阅读开源项目的人。

### 用户能做的事
| 能力 | 说明 |
|------|------|
| 提供仓库路径或 URL | 本地路径或 GitHub 公共仓库 URL |
| 看懂基础 Python 语法 | 单个函数或类可读 |
| 理解文件、目录、函数、类、import | 基础概念已知 |
| 跟随自然语言讲解阅读源码 | 不需要自行规划阅读顺序 |

### 用户不能稳定做到的事
- 判断陌生仓库最值得先理解的点是什么
- 判断某个模块在整体系统里为什么存在
- 沿着真实工程建立入口、分层、主流程与职责拆分认知
- 自己把零散源码事实组织成可迁移的工程理解
- 在多轮追问里维护当前学习进度和下一步学习重点

## <a id="scope"></a>SCOPE — 范围与优先级

### P0：第一版必须完成
```
- 本地仓库路径接入
- GitHub 公共仓库 URL 接入
- 安全只读访问
- 文件树扫描与关键目录识别
- Python 项目画像（类型候选）
- 入口候选识别
- import 来源识别（内部 / 标准库 / 第三方 / 未知）
- 首轮教学产物生成
- 多轮持续教学推进
- 单点展开式讲解
- 下一教学细点引导
- 候选主流程 / 数据流教学骨架
- 模块关系讲解
- 教学式分层理解
- 内容充实度自检
- 多轮上下文保持（≥5 轮）
- 学习目标切换
- 非 Python 仓库降级支持
```

### P1：强烈建议完成
```
- 阶段性总结
- 更稳定的深浅讲解控制
- 更细的设计意图与职责拆分讲解
- 更稳的教学推进策略
- 更稳的大仓库分阶段输出
```

### P2：后续版本考虑
```
- 符号级路径分析
- 更多语言支持
- 私有仓库认证
- 文档混合输入
- 更复杂可视化
- 更完整的上下文管理实验
```

### 明确不支持（第一版）
```
- 私有 GitHub 仓库认证
- 多仓库联合分析
- PDF / 课件 / 设计文档混合输入
- 在线运行代码
- 安装依赖
- 执行 shell 脚本
- 运行测试
- 修改仓库文件
- 自动生成 PR 或 commit
- 把产品做成通用 Agent OS
- 把产品做成多 Agent 编排平台
```

## <a id="input"></a>INPUT — 输入规范

### 仓库输入类型
| 类型 | 格式 | 备注 |
|------|------|------|
| 本地仓库路径 | 绝对路径字符串 | 必须是可读目录 |
| GitHub 公共仓库 URL | `https://github.com/owner/repo` | 必须可公开访问 |

### 输入错误处理
| 错误情况 | 必须输出 |
|----------|----------|
| 路径不存在 | 明确错误说明 |
| 路径不是可读取目录 | 明确错误说明 |
| GitHub URL 不合法 | 明确错误说明 |
| GitHub 仓库不可公开访问 | 明确错误说明 |
| 仓库过大导致首轮无法完整分析 | 明确说明并启用降级教学 |

### 用户追问输入
Agent 必须支持但不限于以下类型的追问：
```
"这个仓库先理解什么最重要？"
"入口到底起什么作用？"
"为什么这里要拆成这几个模块？"
"先不要讲全局，先讲这一个点。"
"这段流程在整体系统里意味着什么？"
"这个模块为什么值得现在看？"
"下一步最值得接着学哪一小块？"
"只看数据库相关部分。"
"讲浅一点。"
"讲深一点。"
```

### 输入安全边界 — 敏感文件黑名单
下列文件命中时，默认不读取正文，只记录“文件存在但未读取内容”：
```
.env
.env.*
*.pem
*.key
*.crt
id_rsa
id_ed25519
credentials*
secrets*
token*
```

## <a id="output"></a>OUTPUT — 教学产物规范

### 总原则
用户可见主产物必须是教学内容，而不是分析报告。

输出规范应约束“讲什么”和“讲到什么程度”，而不是强制“每轮必须按什么栏目排版”。

系统不得把以下内容做成每轮固定外显模块：

- 系统介绍
- 认知框架
- 设计意图
- 阅读路径
- 证据说明
- 不确定项
- 下一步建议

这些内容是教学维度池，不是固定栏目。

### 教学维度池
每轮回答可从以下维度中选择最相关的内容组织教学：

1. 这是什么系统
2. 应先建立什么认知框架
3. 为什么这么设计
4. 阅读路径和教学推进
5. 关键源码锚点

约束：

- 每轮至少把其中一个维度对应的一个具体点讲透
- 可以自然串联其他维度，但不得平均铺开
- 不要求每轮覆盖全部维度
- 不得为了“看起来完整”而做成栏目填空

### 教学单元
单轮输出的最小合格产物是一个完整的教学单元，而不是一组稀薄结论。

一个合格教学单元至少应让用户得到：

- 当前这个点到底是什么
- 它为什么值得现在理解
- 它在整体系统里的位置或意义
- 它在源码里的最小必要落点
- 紧接着最自然该学的下一细点

---

### OUT-1：首轮教学产物 `[P0]`

**触发条件**：首次仓库接入分析成功后必须输出。

**目标**：建立第一层工程认知，并选定一个最值得先讲透的点。

**要求**：

- 首轮可以先给轻量全局地图，但不能停在地图层
- 首轮必须落到一个具体教学点上做详细展开
- 该点可以是入口、结构、分层、主流程、关键模块或设计取舍中的一个
- 该点必须结合少量源码锚点展开
- 首轮结尾必须给出一个紧接当前内容的下一教学细点

**禁止**：

- 只是文件树 dump
- 一次性把所有重要点都各讲两句
- 只罗列证据、候选和置信度
- 用“大而泛”的阅读建议替代当前点的详细讲解

---

### OUT-2：多轮教学推进 `[P0]`

**目标**：像一堂持续推进的课，而不是每轮重启一次解释。

**要求**：

- 每轮优先推进当前最重要的一个知识点或理解缺口
- 若用户切换目标，系统应重新选定新的当前教学点
- 每轮回答不应重新从零介绍仓库
- 已经建立的认知框架应被复用，而不是重复铺垫
- 若当前点尚未讲透，不应急于切换到更多主题

**允许的组织方式**：

- 自然语言连续讲解
- 先结论后解释
- 先问题后带学
- 轻量分段或小标题

**不允许的退化方式**：

- 多点浅讲
- 固定模板化回答
- 证据堆砌压过教学正文

---

### OUT-3：下一教学细点引导 `[P0]`

**定义**：每轮结束时，系统给出的不是“你还可以问什么”的列表，而是一个自然衔接当前内容的下一教学细点提案。

**要求**：

- 必须是单一、具体、可继续讲透的点
- 必须和本轮内容强相关
- 必须说明为什么现在接着讲这个最合适
- 必须让用户大致感知下一轮的学习收益
- 不应同时抛出多个并列方向

**合格示例风格**：

- “如果继续往下，下一步最值得单独讲的是请求进入入口之后，第一层分发到底发生在路由层还是服务层，因为这会把你刚建立的整体结构认知落到真实调用起点上。”

**不合格示例风格**：

- “你还可以继续问入口、模块、流程、依赖、分层。”
- “下一步可以看看别的文件。”

---

### OUT-4：关键源码锚点策略 `[P0]`

**目的**：让教学贴着源码，而不是退化成文件导览。

**要求**：

- 每轮只保留少量必要源码锚点
- 锚点必须直接服务当前教学点
- 锚点可以是文件、目录、函数、类、配置项或 README 段落
- 如果一个点需要过多锚点才能讲清，优先说明焦点可能过散

**禁止**：

- 把“我看过哪些文件”当成正文内容
- 用大量路径替代解释
- 因为源码线索很多就扩散到多个点

---

### OUT-5：候选主流程 / 数据流教学骨架 `[P0]`

**目标**：形成可教学的流程理解，而不是证明真实运行时调用链。

**要求**：

- 当主流程本身就是当前最值得讲的点时，可以围绕它形成单点展开
- 讲解重点应放在“系统怎么动起来”和“为什么要这样拆”
- 不要求覆盖完整运行时细节
- 证据不足时，缩小断言范围后继续教学，不应立即退化为流程不可知

**允许的深度**：

| 深度级别 | 描述 |
|----------|------|
| 入口附近流程 | 只讲入口附近发生了什么 |
| 模块级路径 | 讲清模块之间如何衔接 |
| 教学式数据流骨架 | 讲清输入、经过哪些层、结果去向哪里 |

**原则**：流程讲解是教学内容，不是证据展示。

---

### OUT-6：模块关系与教学式分层 `[P0]`

**目标**：帮助用户建立工程感，而不是追求架构学术严谨。

**要求**：

- 分层和模块关系只有在它们能帮助讲透当前点时才展开
- 可以使用入口层、路由层、业务层、数据层、工具层等启发式分层
- 不得仅凭目录名强行断言
- 若当前仓库无法稳定分层，应明确标注启发式或未知
- 讲分层时必须说明它如何帮助理解当前点

---

### OUT-7：学习目标切换响应 `[P0]`

当用户显式切换学习目标时，系统应：

- 重选当前教学点
- 调整下一教学细点
- 保留仍然有效的既有认知框架
- 不再强行沿旧线继续讲

示例：

- “只看入口” -> 选定入口作为当前单点
- “只看某个模块” -> 选定该模块职责或位置作为当前单点
- “只看分层” -> 选定分层中最影响理解当前系统的那个点
- “讲深一点” -> 保留当前单点，但增加实现细节和设计理由

## <a id="teaching"></a>TEACHING — 教学推进机制

### 单点展开原则
一轮回答的首要目标不是覆盖更多点，而是让用户真正理解当前这一个点。

系统每轮应先判断：

- 当前最该推进的教学点是什么
- 为什么是现在讲这个，而不是别的点
- 这个点应落在哪些最小源码锚点上
- 讲到什么深度才算“讲透”

### 教学点选择优先级
默认优先选择：

1. 能快速建立整体认知的点
2. 能解开当前用户困惑的点
3. 能作为后续多个点前置条件的点
4. 在当前仓库有明确源码锚点支撑的点

### 软推进机制
系统不应被固定模块驱动，而应被教学推进驱动。

软推进的含义：

- 当前只讲一个点
- 但这个点讲得足够细
- 讲完后自然给出下一细点
- 通过多轮把课程推进下去

### “结合源码，但不被源码锁死”策略
教学内容必须由源码锚定，但允许向上抽象到：

- 系统目标
- 设计意图
- 职责拆分
- 架构模式
- 阅读方法

不允许：

- 因为某个细节证据不足，就把整轮回答退化为免责声明
- 只报路径、目录、函数名，不解释其工程含义
- 把源码局部事实误当成教学内容本身

### 内容充实度自检 `[P0]`
这是内部质量机制，不是用户可见模块。

系统在生成回答前后应自检：

- 这轮是否真的讲清了一个点，而不是只命名了这个点
- 是否解释了它在整体系统中的作用，而不是只说它在哪个文件
- 是否给出了“为什么”，而不只是“是什么”
- 是否结合了最小必要源码锚点，而不是脱离仓库空讲
- 是否因为证据不足而过早停止教学，而不是缩小断言后继续讲解
- 是否存在大段保守措辞挤占了教学正文
- 下一步引导是否也是一个足够细的点，而不是泛泛目录

若自检不通过，系统应优先执行以下修正之一：

- 补足解释与设计理由
- 收束到更单一的教学点
- 压缩证据露出
- 增加与整体系统的关系说明
- 把下一步引导改写成更细的下一教学点

## <a id="evidence"></a>EVIDENCE — 证据与不确定性策略

### 定位
证据是教学的支撑层，不是产品主产物。

### 基本规则
- 确定性判断必须有证据
- 无法可靠判断时必须标注未知或推断
- 静态分析不能被表述成真实运行时事实
- 证据露出应遵循最小必要原则

### 候选措辞
当静态分析无法证明真实运行路径时，可使用：
```
候选入口
候选主流程
候选数据流
启发式分层
基于当前证据推断
```

### 置信度使用原则
可以在内部使用或在必要时外显，但不得让置信度成为正文主角。

不合格表现：

- 讲了很多“高/中/低”，却没讲清当前点
- 大量候选措辞把回答压扁

合格表现：

- 先讲清当前点
- 再用少量证据与不确定性约束结论边界

## <a id="ux"></a>UX — 教学体验规则

### 规则 1：教学正文优先
用户首先应读到的是带理解的讲解，而不是分析字段。

### 规则 2：先建立一层认知，再深挖一个点
首轮不应直接堆细节，也不应停在全局地图层。

### 规则 3：每轮只推进一个细点
默认每轮：

- 1 个核心教学点
- 1 条主线
- 1 个下一教学细点

不鼓励：

- 2–4 个并列重点
- 多条主线同时展开

### 规则 4：下一步引导也必须细
下一步引导必须像老师安排的下一小节，而不是“你还可以问什么”的菜单。

### 规则 5：自然语言优先
- 第一版主要输出自然语言讲解
- 小标题、列表、表格只作为辅助
- 不要求固定格式

### 规则 6：连续讲解优先
- 不应频繁插入小测验打断主流程
- 自查问题只作为内部机制，不默认暴露给用户

## <a id="features"></a>FEATURES — 功能需求列表

### F1. 仓库只读接入 `[P0]`
支持本地仓库路径和 GitHub 公共仓库 URL 接入。

### F2. 文件树扫描与关键对象识别 `[P0]`
扫描仓库文件树，识别关键目录和对象。

### F3. 工程认知引导 `[P0]`
在正式深挖前，帮助用户建立最小观察框架并映射到当前仓库。

**验收**：
- [ ] 首轮回答不是通识课，也不是分析结果罗列
- [ ] 首轮回答先建立第一层认知
- [ ] 首轮回答能选出一个最值得先讲透的点

### F4. Python 项目画像 `[P0]`
判断 Python 仓库的项目类型候选。

### F5. 入口候选识别 `[P0]`
识别入口候选，供教学使用。

### F6. import 来源识别 `[P0]`
解释 import 来源，支持内部、标准库、第三方、未知四类分类。

### F7. 首轮教学产物 `[P0]`
首次分析后生成以教学正文为主的首轮产物。

**验收**：
- [ ] 有轻量全局地图
- [ ] 有一个详细展开的当前教学点
- [ ] 有最小必要源码锚点
- [ ] 有一个下一教学细点引导

### F8. 多轮持续教学推进 `[P0]`
围绕同一仓库连续追问时，保持教学连续性。

### F9. 单点展开机制 `[P0]`
保证每轮只讲一个细点，但必须讲透。

### F10. 下一教学细点引导 `[P0]`
每轮结束时生成一个细致的下一教学点，而不是方向列表。

### F11. 候选主流程 / 数据流讲解 `[P0]`
围绕某个入口或路径形成可教学的流程骨架。

### F12. 模块关系与教学式分层 `[P0]`
在必要时帮助用户建立模块关系与分层理解。

### F13. 内容充实度自检 `[P0]`

### F14. 学习目标切换 `[P0]`

### F15. 安全只读策略 `[P0]`

### F16. 阶段性总结 `[P1]`

### F17. 深浅控制 `[P1]`

## <a id="nfr"></a>NFR — 非功能需求

### 准确性
| 规则 | 描述 |
|------|------|
| 证据强制 | 确定性结论必须有证据 |
| 推断标注 | 推断必须标注为推断 |
| 未知输出 | 无法判断时必须输出未知 |
| 候选边界 | 候选流程必须保留候选性质 |

### 教学质量
- 输出长度应主要消耗在教学正文，而不是证据附注
- 单轮必须存在清晰主线
- 回答应以理解收益为目标，而不是信息覆盖率为目标
- 下一教学细点必须可执行、可继续讲透

### 可理解性
- 默认语言面向初学者
- 首轮不堆砌术语
- 解释围绕“它在整体里做什么”“为什么现在看这个”
- 代码细节服务于工程理解，不逐行翻译

### 性能目标
| 仓库规模 | 源码文件数 | 目标 |
|----------|-----------|------|
| 小型 | <500 | 30 秒内开始首轮输出或给进度说明 |
| 中型 | 500–3000 | 90 秒内开始首轮输出或给阶段性结果 |
| 大型 | >3000 | 明确启用降级教学，优先输出第一层认知和第一个教学点 |

### 安全与隐私
```
- 不读取敏感文件正文
- 不输出疑似密钥
- 不执行仓库代码
- GitHub 第一版只支持公开仓库
- 本地仓库访问必须防止越界读取
```

### 可测试性
系统行为必须能通过固定样例仓库验收，至少验证：
```
- 首轮是否形成教学单元
- 多轮是否保持单点推进
- 下一步引导是否为单一细点
- 内容是否避免空泛与保守废话
- 关键判断是否有证据边界
- 安全文件是否未读取正文
```

## <a id="acceptance"></a>ACCEPTANCE — 验收标准

### 样例仓库要求
第一版至少准备 3 个固定测试仓库（固定 commit 或本地 fixture）：

| 编号 | 类型 |
|------|------|
| T1 | 小型 Python CLI 项目 |
| T2 | 小型 Python Web 项目（Flask / FastAPI / Django） |
| T3 | Python library / package 项目 |

每个样例仓库必须预先标注：

- 期望入口候选或允许未知的理由
- 关键目录
- 主要内部模块
- 代表性标准库 import（≥3 个）
- 代表性第三方 import（≥3 个）
- 至少 1 条可接受的候选主流程 / 数据流骨架，或无法形成流程的理由
- 至少 2 个适合作为教学起点的细点

### 首轮教学产物验收
- [ ] 用户首先读到的是教学正文，而不是分析清单
- [ ] 首轮有轻量全局地图，但不会停在地图层
- [ ] 首轮选定了一个具体教学点
- [ ] 这个点被详细展开，而不是只提名称
- [ ] 回答包含最小必要源码锚点
- [ ] 回答结尾给出一个单一的下一教学细点

### 单点展开验收
- [ ] 单轮只有一条清晰主线
- [ ] 不存在多个重点平均分配篇幅
- [ ] 当前点被解释为“是什么 + 为什么重要 + 在整体里做什么 + 源码落点”
- [ ] 讲解足够细，用户能明确知道自己学到了什么

### 下一教学细点验收
- [ ] 每轮只给出一个下一教学点
- [ ] 该点与本轮内容强相关
- [ ] 该点足够具体，不是目录式菜单
- [ ] 该点说明了为什么现在值得接着学
- [ ] 用户能从中看出下一轮预期收益

### 内容充实度验收
- [ ] 不存在只有抽象名词、没有展开解释的段落主导正文
- [ ] 不存在大量“可能 / 候选 / 置信度中等”挤占正文的情况
- [ ] 不存在只有路径和文件名、缺少工程含义解释的情况
- [ ] 不存在因为证据不足就完全停止教学推进的情况

### 多轮状态验收（连续 5 轮）
- [ ] 当前仓库不丢失
- [ ] 当前学习目标不无故变化
- [ ] 当前教学点随用户意图合理切换
- [ ] 已建立的认知框架能被复用
- [ ] 已讲过的点不会每轮从零重复
- [ ] 下一教学细点会随对话推进持续变化

### 学习目标切换验收
- [ ] 切换到“只看入口”后，系统聚焦入口相关单点
- [ ] 切换到“只看依赖来源”后，系统聚焦 import 来源相关单点
- [ ] 切换到“只看分层”后，系统聚焦最关键的分层理解点
- [ ] 切换到“只看某个模块”后，系统聚焦该模块职责或位置
- [ ] 要求“讲浅一点”后，回答更少术语但不变空泛
- [ ] 要求“讲深一点”后，回答更细但仍保持单点展开

### 证据边界验收
以下判断必须有证据边界或明确标注未知：

- [ ] 项目类型
- [ ] 入口候选
- [ ] 关键目录
- [ ] 主要模块职责
- [ ] import 来源
- [ ] 候选主流程步骤
- [ ] 分层映射

但同时：

- [ ] 证据露出不会压过教学正文
- [ ] 不确定性说明不会阻断教学继续进行

### 安全验收
测试仓库中放置 `.env`、`secrets.json`、`id_rsa` 等文件时：

- [ ] 系统不读取正文
- [ ] 输出不包含敏感文件内容
- [ ] 回答仍能在不读取敏感文件的前提下继续教学

## <a id="constraints"></a>CONSTRAINTS — 硬性约束

### 约束 1：不能退化成分析报告生成器
以下现象视为不合格：

- 用户主要看到的是字段、表格、证据和置信度
- 每轮像在填写同一套模板
- 内容覆盖面很广，但没有一个点讲透

### 约束 2：不能退化成抽象空话教学器
以下现象视为不合格：

- 大量宏观判断没有源码落点
- 大量“这是典型分层”“这是常见设计”但没有结合当前仓库
- 只有术语更换，没有真实理解增量

### 约束 3：不能把教学维度写成硬模块
系统内部可以有教学维度池，但产品规格不要求每轮必须以固定栏目外显。

### 约束 4：不能把下一步引导做成菜单
“下一步问什么”必须升级为“下一教学细点提案”。

### 约束 5：必须保持只读安全边界
第一版不得执行仓库代码、不得安装依赖、不得修改仓库文件。
