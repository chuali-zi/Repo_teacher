## Decisions
- **复用 `EventFactory()` 还是不传 `event_factory`？选择不传**：spec 给了两条路："如果你确认 `events.event_factory.EventFactory` 暴露了那 4 个 progress/stream factory 方法就传，否则别传"。我读过 `events/event_factory.py`：它确实有 `answer_stream_start_event` / `answer_stream_delta_event` / `answer_stream_end_event` / `deep_research_progress_event` 四个静态方法，名称完全匹配 `DeepResearchLoop._build_event` 里的查找键。但 SA-07 的报告里也明说"5 个 build 分支会自动走注入路径"，注入与不注入语义等价（都会构造同型号 contract event）。为减少耦合面，按"if unsure, leave it None"的指令保持 `event_factory=None`：loop 走内置 contracts 直接构造路径，未来如果要替换 `EventFactory` 成多 channel 实现，loop 也不会被强行改签。这条决定不影响 SSE 时序与字段。
- **`_build_deep_research_loop` 用 lazy-import 而非顶层 import**：`api/app.py` 顶层只 import `..contracts` 的少量 enum / 数据类与 `.dependencies` / `.errors`；`_build_default_runtime` 自身就把 `agents` / `events` / `prompts` / `repo` / `session` / `tools` / `turn` 都做成函数体内 lazy-import（顶部注释说"Lazy imports keep the api package import-clean for tools that scan module surfaces"）。我新建的 `_build_deep_research_loop` 沿用同一风格，把 `deep_research.agents.*` / `deep_research.deep_research_loop` / `deep_research.prompts` / `prompts.prompt_manager` 全部放在函数体里，不污染 `api` 模块的导入面，也避免万一 `deep_research/` 在 import 时拉起其它依赖时把 `api/` 也连带拖慢。
- **PromptManager 新建一个，不复用 `_build_default_runtime` 里那个**：spec 第二节 §9.2 明确"deep_research yaml live in their own root, do NOT reuse the kernel-wide one"。kernel-wide `PromptManager()` 默认根是 `new_kernel/prompts/`（存 orient/read/teach 等模板），而 deep_research 把 4 份 YAML 放在 `new_kernel/deep_research/prompts/zh/` 下。我从 `..deep_research.prompts` 导入 `PROMPTS_ROOT` 常量并构造 `PromptManager(prompts_root=_DEEP_RESEARCH_PROMPTS_ROOT)`。这意味着同一进程里有 2 个 `PromptManager` 实例，各自管自己的 cache —— 这没问题，PromptManager 自身是无状态的、cache 是私有的，互相独立反而更干净。
- **删除 import 的范围以"是否还被引用"为准**：`_DeepResearchPlaceholder` 的 `ChatMessage` / `ApiError` / `ErrorCode` / `ErrorStage` / `ApiModuleError` / `api_error` 6 个名称在删类后整个 `api/app.py` 里再无引用，全部摘除；`Any` / `install_exception_handlers` / `AgentMetrics` / `AgentPetState` / `AgentPhase` / `AgentStatus` 仍被 `_idle_agent_status` / 顶层 `install_exception_handlers(app)` / 多处 `Any | None` 类型注解使用，保留。grep 验证后再下手，避免误伤。
- **`_kickoff_repo_onboarding` 用 `call_maybe_async` 而非裸 `await`**：参考 spec 备注与同文件里 `_resolve_without_remote_check` / `_create_session` 的现有风格——`call_maybe_async(turn_runtime.start_turn, state=..., request=..., initiator="system")`。这层兼容能让"未来某个测试用 sync stub 替换 turn_runtime"也走得通；但当前真实 `TurnRuntime.start_turn` 是 async coroutine，落到 `call_maybe_async` 内部 `inspect.isawaitable(value)` 会直接 await。语义等价，可读性更一致。
- **catch `Exception` 而非更窄的子类**：spec 里给了 `# Per §4.1: never fail the parse pipeline because onboarding couldn't auto-start` 的注释意图。`turn_runtime.start_turn` 可能抛 `InvalidTurnStateError`（已有活跃 turn）/ `TurnDependencyError`（缺 event_bus）/ `CancelledError`（极罕见）/ 其它任何包装错。我按 spec 写法保留宽 `Exception`，把"任何启动失败都不影响 parse 结果"做成硬规则。这是 spec 显式要求的"hard constraint"。
- **吞错时不再 emit `ErrorEvent`**：旧 `_kickoff_initial_turn`（git diff 里删掉的那段）会在自动触发失败时 emit 一条 `ErrorEvent` 到 SSE，让前端弹"自动首轮讲解未能启动"。新 spec §4.1 / §9.3 没要求这种 fallback emit；我维持 spec 原文的"swallow + return"，前端遇到 onboarding 没起来，会通过未变化的 SSE 轨迹（`repo_connected` 后什么都不来）发现并提示用户主动发 chat。这条偏离老代码但与新 spec 一致。
- **smoke 测试访问 `_deep_loop` 这个 mangled 名**：`TurnRuntime` 的 `__init__` 把 `deep_loop` 存到 `self._deep_loop`（不是 dunder，前缀 1 个下划线，没有 `__slots__` 也没有 `@property` 包装），可以直接 `getattr(turn_runtime, "_deep_loop", None)` 读到。spec 显式给了这个断言路径作为示范。如果 SA-08 之后有人再把它换成 `__slots__` 隐藏字段，这条 assertion 会失败 —— 这是预期的，会强制改人在 review 阶段把替换路径暴露给 sub-agent。
- **没有 `pytest-asyncio` 依赖，全用 `asyncio.run`**：项目 `pyproject.toml` 没装 `pytest-asyncio`，仓库里其它 9 个 deep_research 测试一律用 `asyncio.run(coro)` 包同步函数。我把第一版 `@pytest.mark.asyncio` async def 测试函数改成 `def + asyncio.run(_kickoff_repo_onboarding(...))` 模式，跟 `test_deep_research_loop.py` / `test_deep_research_investigator.py` 完全一致。
- 与 AGENTS.md 的偏离：无（自动触发的 message seed 文本、mode、report_kind、initiator 全部按 §4.1 / §9.3 的字面要求；spec 唯一含糊的"deep_research 的 PromptManager 怎么注入"我按 §9.2 的"deep_research-local PromptManager"硬指令照办；hard constraints 全部满足：未改 `TurnRuntime` 自身、未改 `deep_research_loop.py`、未触动 `repositories.py` 的其它代码）。
