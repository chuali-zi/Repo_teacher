Teaching Deep Research 工作行为目标


（这是一个后端的新模块，因为过大与原有后端分离，后续要接入前端 根目录\web_v3的前端页面，然后llm_config使用根目录的llm config）
一、总体目标

构建一个“教学型深度研究 Agent”，用于对任意代码仓库生成高质量中文教学报告。

该报告必须满足：

内容完整：覆盖架构、模块、实现、数据流
具备教学性：不仅描述，还必须解释“为什么这样设计”
强工程导向：包含优缺点分析与工程实践总结
输出形式：纯中文 + 高质量 Markdown 文本
长度要求：≥3000字（建议 4000–6000）
二、核心行为原则
1. 教学优先（最重要）

所有分析必须转化为“可教学内容”，而不是简单复述代码结构。

每一部分必须包含：

解释设计原因（Why）
说明工程意义（So what）
指出优缺点（Trade-off）
2. 强制结构化表达（但不输出 JSON）

输出必须为结构清晰的 Markdown 文本：

使用标题层级（# / ## / ###）
每个模块结构一致
便于前端自动解析（report map）

禁止：

输出 JSON
输出结构化对象
混合格式
3. 证据驱动分析

所有结论必须基于：

源码
文件结构
import 关系
配置文件


4. 模块语义化（非目录化）

模块划分必须基于“职责”，而不是简单目录：

示例：

API 接入层
会话编排层
模型执行层
文件树索引层

而不是：

backend/m1
backend/m2
5. 强制工程思维输出

每个模块必须包含：

工程习惯
设计模式
常见实践
可迁移经验

旧系统偏“静态分析”
不具备教学能力
架构不支持深度讲解

阶段划分：

research_planning
source_sweep
entry_and_module_analysis
dependency_analysis
chapter_synthesis
final_report_write
3. 模块化 Agent 设计

六、输出规范
必须满足
中文输出
Markdown 格式
≥3000字
层级清晰
可直接阅读
必须包含关键词（用于验收）
“整体架构”
“数据流”
“为什么这样写”
“优点”
“缺点”
“工程习惯”
七、质量验收标准
1. 结构完整性
所有章节存在
每个模块结构完整
2. 教学性
是否解释“为什么”
是否讲清设计取舍
3. 工程性
是否包含工程实践总结
是否具备迁移价值
4. 可读性
Markdown 层级清晰
无 JSON 干扰
5. 长度要求
≥3000字（硬性）
推荐 ≥4000字
八、兼容策略
前端继续使用 raw_text 渲染
后端自动生成 minimal structured_content
模型不参与结构化输出
九、关键设计理念总结
不做“代码总结器”，做“代码老师”
不做“结构描述”，做“设计解释”
不做“目录分析”，做“架构理解”
不做“信息罗列”，做“知识传递”
